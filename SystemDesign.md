## Компоненты структуры сервера
<a id="server-structure-components"></a>
([наверх](#sections))

Капасити (от англ. "capacity") в контексте серверов обычно относится к их вычислительным ресурсам и способности обрабатывать определенный объем данных и запросов. Рассмотрим подробно, какие компоненты обычно входят в структуру сервера, поскольку именно от этих компонентов во многом зависит капасити сервера.

### 1. Процессор (CPU)

- **Описание.** Является мозгом сервера, выполняет инструкции программ и обрабатывает данные.
- **Важные характеристики.** Частота (ГГц), количество ядер и потоков, кэш-память, тепловыделение (TDP), поддержка виртуализации.
- **Влияние на капасити.** Более мощный процессор с большим количеством ядер и высокой частотой может обрабатывать больше задач одновременно, увеличивая капасити сервера.

### 2. Оперативная память (RAM)

- **Описание.** Используется для временного хранения данных и инструкций, активно используемых процессором.
- **Важные характеристики.** Объем (ГБ), тип (например, DDR4), скорость (МГц), тайминги.
- **Влияние на капасити.** Больший объем и более высокая скорость памяти позволяют обрабатывать больше операций и запускать больше приложений одновременно, что напрямую влияет на производительность сервера.

### 3. Хранилище данных (SSD/HDD)

- **Описание.** Используются для долговременного хранения данных.
- **Важные характеристики.** Тип накопителя (SSD для высокой скорости, HDD для большого объема), интерфейс (SATA, SAS, NVMe), скорость чтения/записи, объем хранения.
- **Влияние на капасити.** Высокоскоростные SSD ускоряют загрузку программ и доступ к данным, что положительно сказывается на общей производительности сервера.

### 4. Система охлаждения

- **Описание.** Обеспечивает поддержание оптимальной рабочей температуры компонентов сервера.
- **Важные характеристики.** Тип охлаждения (воздушное, жидкостное), эффективность, уровень шума.
- **Влияние на капасити.** Надежная система охлаждения позволяет компонентам работать на максимальной производительности без перегрева, что важно для поддержания высокой капасити.

### 5. Материнская плата

- **Описание.** Является основой для всех компонентов сервера, обеспечивает их взаимосвязь.
- **Важные характеристики.** Поддерживаемый тип и количество процессоров, слоты для оперативной памяти, типы и количество слотов расширения, интерфейсы для подключения накопителей.
- **Влияние на капасити.** Определяет максимально возможную конфигурацию сервера, его возможности по расширению и модернизации.

### 6. Блок питания

- **Описание.** Обеспечивает электроэнергией все компоненты сервера.
- **Важные характеристики.** Мощность (Вт), эффективность (процентное соотношение входной и выходной мощности), количество и типы разъемов.
- **Влияние на капасити.** Должен обеспечивать достаточную мощность для стабильной работы всех компонентов сервера. Недостаточная мощность может привести к сбоям и остановке работы.

### 7. Сетевой адаптер

- **Описание.** Обеспечивает подключение сервера к сети.
- **Важные характеристики.** Скорость подключения (1 Гбит/с, 10 Гбит/с и выше), тип подключения (медь, оптика), поддержка протоколов и стандартов.
- **Влияние на капасити.** Высокоскоростное подключение к сети позволяет быстро обмениваться данными с другими серверами и клиентами, что критически важно для обслуживания большого числа запросов.

Эти компоненты формируют основу сервера и его способность выполнять вычислительные задачи. От их характеристик зависит общая производительность и капасити сервера. При проектировании серверной инфраструктуры важно учитывать соотношение между потребностями в вычислительных ресурсах и стоимостью оборудования, чтобы обеспечить наиболее эффективное использование ресурсов.

## Типы серверов и их цели
<a id="server-types-and-purposes"></a>
([наверх](#sections))

Существует несколько типов серверов, каждый из которых предназначен для определенных целей и задач. Вот некоторые из наиболее распространенных типов серверов и их основные назначения:

1. **Веб-серверы.** Эти серверы используются для хостинга и обслуживания веб-сайтов. Они обрабатывают HTTP-запросы от клиентских браузеров, отправляя запрошенные веб-страницы и ресурсы. Примеры веб-серверов включают Apache, Nginx, Microsoft IIS и LiteSpeed.

2. **Файловые серверы.** Эти серверы предназначены для хранения и управления файлами и данными. Они позволяют пользователям сохранять, получать и обмениваться файлами через сеть. Примеры файловых серверов включают Windows File Server, NFS (Network File System) и Samba.

3. **Базы данных серверы.** Эти серверы предназначены для хранения и обработки данных, обеспечивая доступ к базам данных для множества пользователей. Они используются для управления и обработки информации, такой как данные клиентов, транзакции и многое другое. Примеры баз данных серверов включают MySQL, PostgreSQL, Microsoft SQL Server и Oracle Database.

4. **Почтовые серверы.** Эти серверы обрабатывают и доставляют электронную почту. Они принимают, обрабатывают и хранят письма, а также обеспечивают протоколы для доступа к электронной почте, такие как POP3, IMAP и SMTP. Примеры почтовых серверов включают Microsoft Exchange Server, Postfix и Sendmail.

5. **Приложительные серверы.** Эти серверы предназначены для выполнения приложений и программного обеспечения для множества пользователей. Они обеспечивают среду выполнения для приложений, обрабатывая запросы и управляя ресурсами. Примеры включают серверы приложений Java EE (например, Apache Tomcat, JBoss), серверы приложений .NET (например, Microsoft IIS с ASP.NET), а также серверы приложений Python, Ruby и других языков программирования.

6. **DNS-серверы.** Эти серверы отвечают за преобразование доменных имен в IP-адреса и наоборот. Они обеспечивают распределение запросов на доменное имя между различными серверами и поддоменами. Примеры DNS-серверов включают BIND (Berkeley Internet Name Domain), Microsoft DNS и Google Cloud DNS.

7. **Хостинговые серверы.** Эти серверы предоставляют услуги хостинга для веб-сайтов и приложений. Они могут включать в себя функции веб-сервера, базы данных, почты и другие сервисы, предоставляя полноценное окружение для хостинга сайтов. Эти серверы могут быть общими (shared hosting), виртуальными (VPS) или выделенными (dedicated hosting).

Каждый из этих типов серверов имеет свои уникальные характеристики и настройки, которые определяются в соответствии с требованиями и целями организации или проекта.

## Влияние архитектуры сервера на капасити и производительность
<a id="impact-of-server-architecture-on-capacity-and-performance"></a>
([наверх](#sections))

Архитектура сервера играет ключевую роль в определении его капасити и производительности. Капасити сервера — это способность обрабатывать определенный объем задач или запросов в единицу времени, не теряя в качестве обслуживания. Эта способность зависит от многих факторов, включая аппаратное обеспечение, программное обеспечение и конфигурацию системы. Вот как различные аспекты архитектуры сервера влияют на его капасити и производительность:

### 1. Центральный процессор (CPU)

- **Мощность и количество ядер.** Более мощные процессоры с большим количеством ядер могут обрабатывать больше инструкций за единицу времени, что напрямую влияет на производительность и капасити сервера. Многозадачность и параллелизм улучшаются за счет использования многоядерных процессоров.
- **Архитектура CPU.** Архитектура, такая как x86-64 или ARM, определяет, как процессор обрабатывает инструкции, что влияет на его эффективность и скорость обработки данных.

### 2. Оперативная память (RAM)

- **Объем и скорость.** Больший объем оперативной памяти позволяет серверу хранить больше данных в непосредственном доступе, уменьшая необходимость обращения к более медленным накопителям. Высокоскоростная RAM ускоряет доступ к данным и обработку, тем самым повышая производительность.
- **Тип памяти.** Современные стандарты, такие как DDR4 и DDR5, предлагают улучшения в скорости передачи данных и энергоэффективности по сравнению с предыдущими поколениями.

### 3. Системы хранения данных

- **Типы накопителей.** SSD (твердотельные накопители) предлагают значительно более высокую скорость чтения/записи по сравнению с традиционными HDD (жесткими дисками), что напрямую влияет на скорость загрузки приложений и доступа к данным.
- **Интерфейсы подключения.** Использование интерфейсов высокой пропускной способности, таких как NVMe, дополнительно ускоряет работу с данными на SSD, улучшая общую производительность системы.

### 4. Сетевая инфраструктура

- **Пропускная способность и задержка.** Высокоскоростные сетевые интерфейсы (например, 10 GbE или выше) уменьшают время передачи данных между сервером и сетью, что особенно важно для высоконагруженных приложений и сервисов, требующих быстрого обмена данными.
- **Технологии виртуализации сети.** Современные технологии, такие как SDN (Software-Defined Networking), позволяют более эффективно управлять сетевыми ресурсами, оптимизируя трафик и уменьшая задержки.

### 5. Архитектура системы и программное обеспечение

- **Оптимизация программного обеспечения.** ПО, оптимизированное под конкретную архитектуру сервера, может значительно улучшить его производительность, например, за счет эффективного распределения задач между ядрами процессора.
- **Виртуализация и контейнеризация.** Технологии, такие как Docker и Kubernetes, а также платформы виртуализации (например, VMware или Hyper-V), позволяют более эффективно использовать ресурсы сервера, улучшая его капасити за счет изоляции и управления нагрузками на уровне программного обеспечения.

### Заключение

Архитектура сервера оказывает комплексное влияние на его капасити и производительность. Баланс между мощностью аппаратного обеспечения и оптимизацией программного обеспечения определяет, насколько эффективно сервер может обрабатывать запросы и выполнять задачи. Правильный выбор компонентов и технологий, а также их настройка под конкретные требования приложений, является ключом к максимизации производительности и капасити сервера.

## Технологии хранения данных для оптимизации капасити и доступности
<a id="data-storage-technologies-for-capacity-and-availability-optimization"></a>
([наверх](#sections))

Для оптимизации емкости (capacity) и доступности данных на серверах используются различные технологии хранения данных. Вот некоторые из них:

1. **RAID (Redundant Array of Independent Disks).**
   - RAID технология объединяет несколько физических дисков в единое логическое устройство для повышения надежности и/или производительности хранения данных.
   - Различные уровни RAID (например, RAID 0, RAID 1, RAID 5, RAID 6) предлагают разные комбинации отказоустойчивости, производительности и емкости.

2. **Хранилища сетевого присоединения (Network Attached Storage, NAS).**
   - NAS представляет собой специализированное устройство для хранения данных, которое подключается к сети и предоставляет хранилище для множества клиентов.
   - NAS обычно обладает высокой емкостью и обеспечивает удобный доступ к данным для различных устройств в сети.

3. **Хранилища подключенных к сети (Storage Area Network, SAN).**
   - SAN - это сеть, объединяющая хранилища данных и серверы с целью предоставления распределенного блока хранения.
   - SAN позволяет эффективно масштабировать хранилище и обеспечивает высокую доступность и производительность.

4. **Облачные хранилища данных.**
   - Облачные хранилища (как публичные, так и частные) предлагают распределенное хранение данных через интернет.
   - Они обычно обладают высокой емкостью, масштабируемостью и гибкостью, обеспечивая доступность данных из любой точки сети.

5. **Дедупликация данных.**
   - Технология дедупликации позволяет уменьшить объем хранимых данных путем удаления дублирующей информации.
   - Это позволяет оптимизировать использование хранилища и уменьшить потребность в дополнительной емкости.

6. **Компрессия данных.**
   - Компрессия данных уменьшает объем хранимых данных путем сжатия информации.
   - Это позволяет сэкономить место на диске и увеличить емкость хранилища.

7. **Технологии тонкого (thin) и толстого (thick) выделения дискового пространства.**
   - Тонкое выделение позволяет выделять меньший объем физического дискового пространства на начальном этапе, увеличивая его по мере необходимости.
   - Толстое выделение резервирует всю заранее указанную емкость сразу, что может привести к неэффективному использованию пространства.

Эти технологии позволяют оптимизировать емкость и доступность данных на серверах в соответствии с требованиями конкретной системы хранения.

## RAID
<a id="RAID"></a>
([наверх](#sections))

RAID (Redundant Array of Independent Disks) – это технология, которая объединяет несколько физических дисков в единое логическое устройство для увеличения надежности, производительности и/или емкости хранения данных. Различные уровни RAID предлагают разные комбинации этих характеристик, обеспечивая оптимальное решение в зависимости от требований конкретного применения. Вот подробный обзор основных уровней RAID:

1. **RAID 0 (Stripe).**
   - RAID 0 разделяет данные на части (блоки) и записывает их на разные диски в наборе.
   - Позволяет повысить производительность, так как операции чтения и записи могут выполняться параллельно на нескольких дисках.
   - Не обеспечивает отказоустойчивости, потому что отказ одного диска приводит к потере всех данных в массиве.

2. **RAID 1 (Mirroring).**
   - RAID 1 создает точную копию данных на двух или более дисках.
   - Обеспечивает высокую отказоустойчивость, так как данные сохраняются даже при отказе одного диска.
   - Жертвует половиной доступной емкости на дублирование данных.

3. **RAID 5.**
   - RAID 5 использует метод распределения данных и паритетный блок для обеспечения отказоустойчивости и повышения производительности.
   - Данные разбиваются на блоки и записываются на несколько дисков, а паритетные блоки вычисляются и распределяются по всем дискам.
   - При отказе одного диска данные можно восстановить из паритетной информации и данных на оставшихся дисках.
   - RAID 5 обеспечивает более высокую емкость по сравнению с RAID 1 за счет использования только одного диска для паритета.

4. **RAID 6.**
   - RAID 6 похож на RAID 5, но вместо одного паритетного блока использует два независимых блока паритета.
   - Обеспечивает еще более высокую отказоустойчивость, так как может выдержать отказ двух дисков.
   - Требует больше вычислительных ресурсов для вычисления и записи паритетных блоков, что может снизить производительность.

5. **RAID 10 (1+0).**
   - RAID 10 объединяет в себе преимущества RAID 1 и RAID 0.
   - Данные зеркалируются на паре дисков, а затем блоки данных страйпуются по этим парам.
   - Обеспечивает высокую производительность и отказоустойчивость, но требует удвоенной емкости по сравнению с простым RAID 0.

Каждый уровень RAID имеет свои преимущества и недостатки, и выбор конкретного уровня зависит от требований к надежности, производительности и емкости хранения данных.

## Методы повышения надежности и капасити серверов
<a id="methods-for-improving-reliability-and-capacity-of-servers"></a>
([наверх](#sections))

### 🔧 Что такое надёжность и капасити

**Надёжность (Reliability)** — способность системы работать корректно в течение заданного времени, несмотря на отказы компонентов.
**Капасити (Capacity)** — максимальная нагрузка, которую система может выдержать без деградации SLA/SLI

Эти два свойства — краеугольные камни в проектировании высоконагруженных систем. Их обеспечение требует как архитектурных решений, так и грамотного использования технологий

### 🛠 Методы повышения надёжности

#### 1. **Репликация (Replication)**

Репликация — это создание копий данных на нескольких узлах.

* **Master-Slave**: Один узел (мастер) — источник истины, остальные — реплики
* **Multi-Master**: Все узлы могут принимать запись (повышенная сложность консистентности)

**Пример (PostgreSQL Streaming Replication):**

```bash
# На мастере
wal_level = replica
max_wal_senders = 3

# На слейве
standby_mode = on
primary_conninfo = 'host=master_ip user=replication password=secret'
```

**Плюсы:**

* Фейловер при отказе
* Горизонтальное масштабирование на чтение

**Минусы:**

* Потенциальная несогласованность при Async репликации

#### 2. **Фейловер и Heartbeat-механизмы**

Фейловер — автоматическое переключение трафика на резервную ноду при отказе основной

**Пример (Keepalived + HAProxy):**

```bash
vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    virtual_ipaddress {
        192.168.1.100
    }
}
```

**Инструменты:**

* Keepalived / Corosync / Pacemaker
* Cloud Load Balancer (GCP, AWS ELB)

#### 3. **Идемпотентность и ретраи**

Все операции должны быть идемпотентны, особенно в сетевом взаимодействии. Повтор одного и того же запроса не должен приводить к ошибке или дублирующим результатам

**Пример (идемпотентная оплата):**

```python
def process_payment(payment_id: str, amount: float):
    if db.payment_exists(payment_id):
        return "Already processed"
    db.create_payment(payment_id, amount)
    return "OK"
```

**Best practices:**

* Использовать UUID для `idempotency key`
* Хранить результат выполнения операции

#### 4. **Circuit Breaker**

Защита от каскадных сбоев. При превышении порога ошибок — выключает обращение к проблемному сервису

**Пример (Python + pybreaker):**

```python
import pybreaker

breaker = pybreaker.CircuitBreaker(fail_max=5, reset_timeout=60)

@breaker
def call_service():
    return requests.get('https://external.api')
```

#### 5. **Load Balancing с health-check**

Нагрузку следует равномерно распределять между серверами. Используется также для изоляции больных нод

**Пример (Nginx):**

```nginx
upstream backend {
    server backend1.example.com max_fails=3 fail_timeout=30s;
    server backend2.example.com;
}
```

### ⚙ Методы повышения капасити

#### 1. **Горизонтальное масштабирование (Scaling Out)**

Банальное добавление новых узлов в кластер

**Пример (Kubernetes Deployment):**

```yaml
spec:
  replicas: 5
```

**Когда применять:**

* Когда отдельный сервер упирается в CPU/Memory
* Если у вас stateless-сервисы

#### 2. **Шардирование (Sharding)**

Разделение данных между несколькими хранилищами по ключу (например, `user_id % N`)

**Пример:**

```python
def get_db_for_user(user_id):
    shard = user_id % 4
    return f"db_shard_{shard}"
```

**Варианты:**

* Range sharding
* Hash sharding
* Geo sharding

**Минусы:**

* Сложность миграции
* Потеря агрегированных запросов по всем данным

#### 3. **Кеширование (Caching)**

Снижение нагрузки за счёт хранения часто используемых данных в памяти

**Виды:**

* Local (внутри приложения)
* Distributed (Redis, Memcached)
* CDN (статические файлы, фронт-контент)

**Пример (Redis):**

```python
import redis

cache = redis.Redis()
value = cache.get("user_123")
if not value:
    value = db.query_user(123)
    cache.set("user_123", value, ex=300)
```

#### 4. **Backpressure и Rate Limiting**

Контроль нагрузки и отказ от обработки запросов сверх лимита

**Пример (Nginx):**

```nginx
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

server {
    location /api/ {
        limit_req zone=api burst=20;
    }
}
```

#### 5. **Асинхронная обработка (Task queues)**

Перенос тяжёлых операций в фон с помощью очередей задач

**Инструменты:** Celery, Kafka, RabbitMQ

**Пример (Celery):**

```python
@app.task
def send_email(user_id):
    # time.sleep(5)
    ...
```

#### 6. **Оптимизация I/O и использования CPU**

* Асинхронные фреймворки (`FastAPI`, `aiohttp`)
* Профилирование (`py-spy`, `perf`, `strace`)
* Использование буферизации, батчинга

**Пример (Batch Write):**

```python
# вместо одиночных insert:
db.bulk_insert(users)
```


## Масштабирование серверных кластеров
<a id="scaling-server-clusters-for-capacity-increase"></a>
([наверх](#sections))

Масштабирование серверных кластеров для увеличения капасити — это комплексный процесс, который включает в себя несколько ключевых этапов и рассматривает различные аспекты, от аппаратного обеспечения до программного обеспечения и архитектуры. Под капасити понимаются ресурсы, необходимые для обработки текущей нагрузки, такие как вычислительная мощность, объем памяти, сетевая пропускная способность и место для хранения данных. Вот как можно подробно расписать процесс масштабирования:

### 1. Анализ текущего состояния

- **Оценка нагрузки.** Измерение текущей нагрузки и прогнозирование будущих требований. Это включает анализ пиковых и средних значений нагрузки, а также учет возможного роста или сезонных колебаний.
- **Аудит ресурсов.** Определение текущего использования ресурсов, включая процессоры, память, хранилище и сеть.
- **Идентификация узких мест.** Выявление компонентов системы, ограничивающих производительность, например, медленное хранилище или недостаточная сетевая пропускная способность.

### 2. Планирование масштабирования

- **Выбор стратегии масштабирования.** Горизонтальное (добавление единиц однотипного оборудования) против вертикального (увеличение мощности существующего оборудования) масштабирования. Горизонтальное масштабирование чаще всего предпочтительнее из-за лучшей отказоустойчивости и масштабируемости.
- **Разработка архитектуры.** Создание или оптимизация архитектуры для поддержки масштабируемости, включая балансировку нагрузки, кластеризацию и репликацию данных.
- **Планирование ресурсов.** Определение необходимых ресурсов, включая аппаратное обеспечение и программное обеспечение, а также логистику их распределения.

### 3. Реализация масштабирования

- **Приобретение и установка оборудования.** Покупка необходимого аппаратного обеспечения и его физическая установка в дата-центре.
- **Настройка и тестирование.** Конфигурирование нового оборудования, установка необходимого программного обеспечения, тестирование производительности и отказоустойчивости.
- **Оптимизация программного обеспечения.** Внесение изменений в программное обеспечение для эффективного использования новых ресурсов, включая оптимизацию баз данных, кэширование и балансировку нагрузки.

### 4. Мониторинг и оптимизация

- **Мониторинг производительности.** Постоянное наблюдение за показателями производительности и использования ресурсов для выявления потенциальных проблем.
- **Анализ данных мониторинга.** Анализ собранных данных для идентификации трендов, узких мест и потребностей в дополнительном масштабировании.
- **Оптимизация.** Внедрение изменений для улучшения производительности и эффективности, включая настройку конфигураций и алгоритмов распределения нагрузки.

### 5. Управление изменениями и документирование

- **Управление изменениями.** Внедрение процесса управления изменениями для обеспечения бесперебойной работы системы в процессе масштабирования.
- **Документирование.** Создание документации по новой инфраструктуре, включая схемы, конфигурации и процедуры восстановления после сбоев.

### 6. Подготовка к будущему масштабированию

- **Разработка стратегии масштабируемости.** Планирование дальнейших шагов масштабирования в соответствии с прогнозируемым ростом и развитием технологий.
- **Обучение персонала.** Повышение квалификации команды для работы с масштабированной инфраструктурой.

## Сетевые технологии для оптимизации капасити и производительности
<a id="network-technologies-for-capacity-and-performance-optimization"></a>
([наверх](#sections))

Оптимизация капасити и производительности серверов часто осуществляется с использованием различных сетевых технологий. Вот некоторые из них:

1. **Load Balancing (Балансировка нагрузки).**
   - **Hardware Load Balancers (аппаратные балансировщики нагрузки).** Эти устройства распределяют сетевой трафик между несколькими серверами для обеспечения равномерного распределения нагрузки и предотвращения перегрузок.
   - **Software Load Balancers (программные балансировщики нагрузки).** Решения на основе программного обеспечения, которые выполняют ту же функцию, что и аппаратные балансировщики, но работают на обычных серверах.

2. **Content Delivery Networks (CDN) (Сети доставки контента).**
   - CDN предоставляют кэширование контента на серверах, размещенных ближе к конечным пользователям. Это уменьшает нагрузку на центральные серверы и ускоряет доставку контента.

3. **Caching (Кэширование).**
   - Системы кэширования временно хранят копии данных, уменьшая необходимость сервера каждый раз обращаться к базе данных или другим источникам данных. Это значительно увеличивает скорость ответа сервера и снижает нагрузку.

4. **Network Accelerators (Сетевые ускорители).**
   - Эти устройства оптимизируют сетевую передачу данных, используя техники, такие как сжатие данных, управление потоком и предварительное получение данных, чтобы улучшить производительность сети и снизить задержки.

5. **Quality of Service (QoS) (Качество обслуживания).**
   - QoS технологии позволяют определять приоритеты для различных типов сетевого трафика. Это позволяет оптимизировать использование пропускной способности сети и гарантировать достаточные ресурсы для критически важных приложений.

6. **Multiprotocol Label Switching (MPLS) (Многопротокольная меткосменная коммутация).**
   - MPLS позволяет оптимизировать маршрутизацию трафика в сети, улучшая производительность и предоставляя гарантированную пропускную способность для приложений с высокими требованиями к качеству обслуживания.

7. **Software-defined Networking (SDN) (Сети, управляемые программным обеспечением).**
   - SDN позволяет централизованно управлять сетью и динамически настраивать её параметры в соответствии с текущей нагрузкой и требованиями приложений, что способствует оптимизации капасити и производительности.

8. **Virtualization (Виртуализация).**
   - Технологии виртуализации позволяют эффективно использовать вычислительные ресурсы серверов путем создания виртуальных экземпляров операционных систем и приложений. Это упрощает масштабирование приложений и повышает гибкость инфраструктуры.

## Техники охлаждения в серверных центрах
<a id="cooling-techniques-in-server-facilities-for-optimal-operation-and-capacity"></a>
([наверх](#sections))

В серверных центрах применяются различные техники охлаждения для поддержания оптимальной работы оборудования и обеспечения достаточной капасити. Ниже перечислены некоторые из основных методов охлаждения, которые могут использоваться:

1. **Воздушное охлаждение.** Это один из наиболее распространенных способов охлаждения серверных центров. Воздушные кондиционеры или вентиляторы распределяют прохладный воздух по рабочим зонам, чтобы охлаждать оборудование. Системы вентиляции могут быть настроены для оптимального распределения потока воздуха в соответствии с тепловым распределением в помещении.

2. **Жидкостное охлаждение.** Этот метод охлаждения использует жидкость, например, воду или специальные охладители, чтобы отводить тепло от серверов. Жидкость циркулирует через специальные системы трубопроводов, которые проходят мимо оборудования, забирая тепло, а затем выводится в систему охлаждения.

3. **Иммерсионное охлаждение.** Этот относительно новый метод включает погружение серверов в специальные емкости с охладительной жидкостью, которая полностью окружает оборудование. Это позволяет эффективно отводить тепло, так как жидкость непосредственно контактирует с нагревающимися компонентами серверов.

4. **Тепловые насосы.** Этот метод использует тепловые насосы для переноса тепла изнутри серверного центра наружу. Тепловые насосы работают на принципе теплового обмена и могут быть эффективными при правильной настройке.

5. **Адиабатическое охлаждение.** Этот метод использует процесс испарения воды для охлаждения воздуха в серверном помещении. Вода испаряется в воздух, что приводит к снижению его температуры. Этот процесс может быть эффективным в регионах с низкой влажностью.

6. **Теплоотводящие материалы и конструкции.** Кроме технических систем охлаждения, также важно использовать теплоотводящие материалы и конструкции, чтобы уменьшить тепловую нагрузку на серверный центр. Это может включать в себя улучшенные системы утепления, специальные материалы для стоек и корпусов серверов и оптимизированные конструкции для улучшения циркуляции воздуха.

## Методы управления энергопотреблением в целях оптимизации капасити и экономии ресурсов
<a id="energy-management-methods-for-capacity-optimization-and-resource-saving"></a>
([наверх](#sections))

### Введение

Энергопотребление становится критическим аспектом проектирования систем не только из-за стоимости, но и из-за ограничений по охлаждению, размещению оборудования и экологических требований (например, NetZero, ESG).

**Цель управления энергопотреблением** — снизить расходы и улучшить эффективность использования вычислительных ресурсов **без деградации производительности и надёжности**.

### Основные подходы

#### 1. **Dynamic Voltage and Frequency Scaling (DVFS)**

Механизм, при котором ЦП/ГП ускоряются или замедляются путём изменения напряжения и частоты.

**Формула:**

Энергопотребление процессора приблизительно пропорционально:

$$
P \propto C \cdot V^2 \cdot f
$$

где:

* $P$ — мощность,
* $C$ — ёмкость (постоянная для процессора),
* $V$ — напряжение,
* $f$ — тактовая частота.

**Следствие:**
Снижение частоты и напряжения — **квадратичное** снижение энергопотребления.

**Пример (Linux, cpufreq):**

```bash
# Проверить текущую частоту
cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_cur_freq

# Установить governor (режим управления частотой)
echo "powersave" > /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
```

#### 2. **Power-aware workload scheduling**

Перенос задач между серверами в зависимости от энергопрофиля.
Например, активные задачи отдаются узлам с **энергоэффективной архитектурой** (ARM), а не x86.

**Пример (Kubernetes power-aware scheduling):**

* Node labels:

  ```yaml
  node-role.kubernetes.io/energy-efficient: "true"
  ```
* Scheduling constraints:

  ```yaml
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: node-role.kubernetes.io/energy-efficient
                operator: In
                values:
                  - "true"
  ```

#### 3. **Auto-scaling и Load Shedding**

##### 3.1 Horizontal Pod Autoscaling (HPA)

Автоматическое масштабирование на основе метрик (CPU, custom, energy-aware).

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 60
```

##### 3.2 Load Shedding — отказ от некритичных задач при перегрузке

**Пример:**

```python
if cpu_load > 80:
    reject_low_priority_requests()
```

#### 4. **Turn-off / Deep Sleep (C-states, T-states)**

Сервер может входить в глубокий спящий режим, когда не используется.
Используется в масштабных кластерах и облачных провайдерах.

**Intel C-states:**

* C0 — активный
* C1, C2, … — прогрессивно более "глубокий" сон

**Пример настройки (Linux):**

```bash
# Запретить переход в глубокие состояния
echo 1 > /sys/devices/system/cpu/intel_idle/max_cstate
```

#### 5. **Energy-aware programming**

##### 5.1 Минимизация I/O

I/O-операции — очень энергозатратны (особенно на SSD и HDD).

**Пример (Python, write batching):**

```python
# Вместо записи по одному
with open("log.txt", "w") as f:
    for line in lines:
        f.write(line)

# Лучше — записывать пачкой
with open("log.txt", "w") as f:
    f.writelines(lines)
```

##### 5.2 Алгоритмы с энергоэффективной сложностью

Низкое энергопотребление напрямую связано с меньшей вычислительной сложностью.

**Пример:**

* Алгоритм с $O(n)$ предпочтительнее $O(n \log n)$, если точность допустима.
* Пример применения bloom filter для предикативной фильтрации:

```python
from pybloom_live import BloomFilter
bloom = BloomFilter(capacity=100000, error_rate=0.001)
```

### Метрики энергопотребления и оптимизации

* **PUE (Power Usage Effectiveness)**:

  $$
  \text{PUE} = \frac{\text{Total Facility Power}}{\text{IT Equipment Power}}
  $$

* **Watt per request** или **Joules per inference** (в ML):

  $$
  \text{Efficiency} = \frac{\text{Energy Used}}{\text{of Successful Operations}}
  $$

* **Node-level metering** — с помощью IPMI:

ipmitool sensor | grep -i power

## Влияние физической конфигурации на капасити и производительность сервера
<a id="impact-of-physical-configuration-on-server-capacity-and-performance"></a>


Что такое капасити сервера в формальном смысле

Под капасити обычно имеют в виду максимально устойчивый поток запросов, при котором система еще держит свои SLO по латентности и ошибкам. Если обозначить

* `RPS_cpu` как предел по CPU
* `RPS_ram` как предел по памяти
* `RPS_disk` как предел по диску
* `RPS_net` как предел по сети

то общий предел можно записать так

$$
RPS_{max} \approx \min(RPS_{cpu}, RPS_{ram}, RPS_{disk}, RPS_{net}) \cdot k_{safety}
$$

где `k_safety` находится где-то в районе 0.6-0.8. Это запас, который оставляют под всплески, GC, репликацию и прочий фон.

Физическая конфигурация сервера задает каждый из этих RPS по отдельности. Дальше разбор по ресурсам.

**CPU и ядра как кассы в магазине**

Удобная аналогия - супермаркет. Ядра процессора - это кассы, поток запросов - это поток покупателей, одно обслуживание на кассе - это обработка одного запроса.

Пусть у нас есть процессор с `N_cores` ядрами. Одно ядро работает на частоте `f` ГГц, выполняет в среднем `IPC` полезных инструкций за такт, а один запрос требует `I_req` инструкций.

Тогда теоретический предел по CPU можно оценить как

$$
RPS_{cpu} \approx \frac{N_{cores} \cdot f \cdot 10^9 \cdot IPC}{I_{req}}
$$

Пример. API на Java обрабатывает около `5 · 10^6` инструкций на запрос (цифра условная, порядок величины). Процессор имеет 16 ядер, каждая с эффективной all-core частотой 3.5 ГГц и IPC около 2. Тогда

$$
RPS_{cpu} \approx \frac{16 \cdot 3.5 \cdot 10^9 \cdot 2}{5 \cdot 10^6} \approx \frac{112 \cdot 10^9}{5 \cdot 10^6} \approx 22400\ RPS
$$

Это абстрактная верхняя оценка без учета блокировок, контекстных переключений, кэшей и так далее. В реальности берут от половины до трети.

**Hyper-Threading как две очереди к одной кассе**

Hyper-Threading можно представить как кассира, у которого два "потока" покупателей, которые он обслуживает вперемешку. Когда один достает карту, второй уже выкладывает продукты. При IO-bound нагрузке это помогает заполнить простои и дает добавку к RPS. При жестком CPU-bound прирост меньше, иногда деградация по латентности из-за конкуренции за кэш.

**NUMA и кеши как два склада**

В многосокетной машине у каждого сокета свой "склад" памяти. Когда поток на первом сокете лезет за данными на склад второго, время доступа растет. В модели можно ввести `L_local` и `L_remote` как латентность локальной и удаленной памяти. Средняя латентность тогда

$$
L_{avg} = p_{local} \cdot L_{local} + p_{remote} \cdot L_{remote}
$$

При малой доле удаленных обращений `p_remote` изменение почти не заметно. При перекосе планировщика, когда процесс постоянно прыгает между сокетами, `p_remote` растет и приложение внезапно "тормозит", хотя мониторинг CPU показывает свободные ядра.

Жизненный пример. Двухсокетный сервер с PostgreSQL без настройки NUMA. На холодном старте все красиво. Через несколько часов под нагрузкой pgсlass и pgstat лежат в удаленной памяти, планы запросов с join начинают ехать по латентности в разы, разработчики подозревают "кривые индексы", а проблема в том, что процесс и память оказались на разных нодах.

**Память как склад с полками**

ОЗУ определяет размер "горячего" набора данных, который можно держать рядом с CPU. Если представлять систему как магазин, то память - это стеллажи, куда выкладывают ходовой товар. Диск - это удаленный склад в соседнем районе.

Пусть

* `W_set` объем рабочего набора данных
* `RAM` общий объем доступной памяти под данные
* `p_mem` доля обращений, попадающих в RAM
* `p_disk` доля обращений, которые уходят на диск

Тогда можно оценить среднюю латентность чтения данных

$$
L_{data} = p_{mem} \cdot L_{mem} + p_{disk} \cdot L_{disk}
$$

При этом `L_mem` обычно в десятки наносекунд, `L_disk` для NVMe в десятки или сотни микросекунд, для HDD в миллисекунды.

Как только `W_set` перестает помещаться в `RAM`, доля `p_disk` растет с примерно нуля до значимых процентов. Средняя латентность прыгает на порядки, RPS по факту падает.

Простой пример с Redis и PostgreSQL.

1. В Redis хватает памяти на кеш ключевых агрегатов. Попадание в кеш на уровне 90 процентов. Большая часть запросов к API укладывается в 10 миллисекунд, база получает только сложные запросы.
2. Трафик вырос, данных стало больше, кеш заполняется, Redis начинает активно вытеснять ключи. Хитрейт падает до 50 процентов. База внезапно получает вдвое больше запросов, диск начинает дергаться чаще, latency среднего запроса растет с 10 до 80 миллисекунд.

Капасити по RPS в этой точке ограничивается объемом RAM на уровне Redis и PostgreSQL, хотя CPU на серверах еще "дышит".

**Пропускная способность памяти**

Помимо объема у памяти есть пропускная способность, то есть сколько гигабайт в секунду можно перелить между CPU и RAM. Важно количество каналов.

Для задач, которые интенсивно сканируют большие объемы (OLAP, Spark, ClickHouse, in-memory аналитика), предел по RPS можно оценить через пропускную способность. Пусть один запрос читает в среднем `B_req` байт из оперативки, а общая пропускная способность памяти `BW_mem` байт в секунду. Тогда

$$
RPS_{ram} \approx \frac{BW_{mem}}{B_{req}}
$$

Берем пример. Сервер Spark с пропускной способностью памяти 100 ГБ/с. Один запрос аналитики суммарно гоняет 5 ГБ данных по цепочке оператора. Тогда верхняя граница около 20 запросов в секунду. Хоть CPU и 64 ядра, законы физики в памяти не даст обойти.

**Диск и I/O как грузчики и фуры**

Диск похож на погрузку и разгрузку фур на складе.

Есть два ключевых параметра:

* `IOPS` максимальное количество операций ввода-вывода в секунду
* `BW_disk` пропускная способность по данным в секунду

Пусть на один запрос приходится `n_io` операций ввода-вывода и `B_io` байт данных. Тогда оценка будет такой

$$
RPS_{disk_iops} \approx \frac{IOPS}{n_{io}},
\quad
RPS_{disk_bw} \approx \frac{BW_{disk}}{B_{io}}
$$

Реальный предел по диску будет примерно минимумом из этих двух.

Жизненный пример. PostgreSQL крутится на HDD с IOPS порядка 200 случайных операций. Один SELECT по индексу в среднем делает 4 случайных чтения блоков. При этом

$$
RPS_{disk} \approx \frac{200}{4} \approx 50\ запросов\ в\ секунду
$$

После переноса на NVMe c IOPS 50 000 и сравнимым профилем запросов, тот же сервер теоретически выдерживает

$$
RPS_{disk} \approx \frac{50\ 000}{4} \approx 12\ 500\ запросов\ в\ секунду
$$

Понятно, что дальше уже будут ограничения по CPU и сети, но порядок изменения потолка очевиден.

**Сеть как лифты в бизнес-центре**

Сеть можно представить как набор лифтов между этажами. Есть общий лимит по людям в минуту и есть ограничение по размеру кабины.

Пусть `BW_net` общая пропускная способность сети в байтах в секунду, а средний запрос гоняет `B_req_net` байт по сети (request + response + межсервисный трафик). Тогда

$$
RPS_{net} \approx \frac{BW_{net}}{B_{req_net}}
$$

Если сервер стоит за 1Gбит линком, то это примерно 125 МБ/с полезных данных. При среднем запросе 25 КБ потолок по сети около

$$
RPS_{net} \approx \frac{125 \cdot 10^6}{25 \cdot 10^3} \approx 5000\ RPS
$$

Даже если CPU по расчетам тянет 20 тысяч RPS, конкретный сервер упирается в сетевой интерфейс при реальной нагрузке. Это частая история у жирных JVM сервисов, которые гоняют тяжелые JSON между микросервисами.

Второй аспект сети - количество пакетов в секунду. Мелкие запросы с маленькими именами ключей в кэше могут упереться в лимит по pps задолго до байтового лимита. Тогда формула уже идет от `PPS_max / packets_per_request`.

**Виртуализация и noisy neighbor как соседи по офису**

Голое железо дает довольно предсказуемые границы. Как только поверх включают виртуализацию или контейнеры, конфигурация превращается в бизнес-центр, где несколько компаний делят лифты, парковку и кондиционер.

Формально можно описать так. Пусть у физического хоста есть CPU емкостью `C_cpu`, диск `C_disk`, сеть `C_net`. На нем крутится `k` виртуалок или групп контейнеров, каждая хочет свой RPS. Когда суммарный запрос к ресурсу превышает `C_cpu` или `C_disk`, начинается конкуренция.

Верхняя граница для одной ВМ тогда становится функцией от поведения соседей

$$
RPS_{vm} \le f(C_{cpu}, C_{disk}, C_{net}, load_{neighbors})
$$

Когда соседи запускают бэкап или тяжелую analytics job, твой сервис с просчитанным по CPU и RAM потолком внезапно начинает выбрасывать 5xx, хотя задержки внутри приложения не сильно изменились. Лимит находится в диске или сети на уровне хоста.

**Как собрать все в одну модель для system design**

Если дать этому весь вид конспекта базы знаний, то можно предложить простой пошаговый алгоритм.

Сначала описываем "единицу работы". Например, один HTTP запрос на сервис. Для него измеряем или оцениваем:

* `T_cpu_req` среднее время работы на CPU
* `B_mem_req` объем данных, которые приходятся на запрос в оперативке
* `n_io_req` количество операций ввода-вывода
* `B_io_req` суммарный объем этих операций
* `B_net_req` трафик по сети

Дальше для конкретной физической конфигурации сервера известны

* `C_cpu` доступная емкость CPU в секундах работы в секунду, по сути `N_cores`
* `BW_mem` пропускная способность памяти
* `IOPS` и `BW_disk` по дискам
* `BW_net` по сети

Тогда можно приблизительно оценить

$$
RPS_{cpu} \approx \frac{C_{cpu}}{T_{cpu_req}}
$$
$$
RPS_{ram} \approx \frac{BW_{mem}}{B_{mem_req}}
$$
$$
RPS_{disk} \approx \min\left(\frac{IOPS}{n_{io_req}}, \frac{BW_{disk}}{B_{io_req}}\right)
$$
$$
RPS_{net} \approx \frac{BW_{net}}{B_{net_req}}
$$

И уже после этого

$$
RPS_{max} \approx \min(RPS_{cpu}, RPS_{ram}, RPS_{disk}, RPS_{net}) \cdot k_{safety}
$$

Для system design это удобно тем, что можно буквально переигрывать конфигурации "что будет, если". Например, замена HDD на NVMe резко увеличивает `IOPS` и `BW_disk`, соответственно `RPS_{disk}`. Добавление памяти увеличивает `RPS_{ram}` за счет уменьшения `B_mem_req` на счет вероятности очередного доступа к диску. Переход на 10G сеть увеличивает `BW_net` и так далее.

**Пара жизненных сценариев, которые хорошо показывают влияние железа**

Сценарий про "один и тот же код, разные сервера".

Есть микросервис на JVM, который обслуживает примерно 500 RPS на старом сервере:

* 8 ядер по 2.4 ГГц
* 32 ГБ RAM
* HDD под логами и локальными файлами
* 1G сеть

Под нагрузочным тестом латентность 95 перцентиля около 180 миллисекунд, CPU периодически в пике, диск живет своей жизнью с 70 процентами busy time.

Тот же сервис переносят на машину:

* 16 ядер по 3.2 ГГц
* 64 ГБ RAM
* NVMe под временные файлы и логи
* 10G сеть

После короткого тюнинга (heap, GC, thread pool) под тем же нагрузочным профилем:

* CPU в среднем 40 процентов вместо 80
* latency p95 падает до 60 миллисекунд
* потолок по RPS поднимается с 500 до примерно 2 000
* дальше рост ограничивает уже внешняя база данных, у которой свои пределы по диску и сети

Таким образом, физическая конфигурация сервера сдвигает правее точку излома на графике "RPS - latency". Тот же код, те же алгоритмы, но другая железка меняет капасити в разы.

Второй сценарий. PostgreSQL и увеличение RAM.

На стартовой конфигурации:

* RAM 16 ГБ
* рабочий набор данных около 20 ГБ
* диск SSD SATA

В дневной пик запросы к горячим таблицам постоянно ходят на диск, latency по сложным SELECT около 200 миллисекунд во время пикового RPS. После перераспределения ролей и увеличения RAM до 64 ГБ рабочий набор с запасом помещается в памяти. SSD трогают только бэкапы и редкие полный скан. Под тем же RPS:

* latency по SELECT падает до 20-30 миллисекунд
* CPU падает с 90 до 50 процентов
* капасити по RPS вырастает в несколько раз, пока другим потолком не становится CPU или сеть

С точки зрения формул это увеличение `p_mem` и уменьшение `p_disk` в выражении для `L_data`, из-за чего общая латентность резко падает, а доступный `RPS_max` увеличивается.

### Как провести оценку

1. **Сбор данных.** Используем инструменты мониторинга и аналитики, такие как Prometheus, Nagios, Zabbix или другие системы для сбора вышеупомянутых метрик в реальном времени.

2. **Анализ нагрузки.** Проанализируем полученные данные, чтобы понять, как нагрузка распределяется в течение рабочего дня или недели. Важно определить пиковые и спадные периоды использования.

3. **Прогнозирование.** На основе исторических данных о загрузке серверов можно прогнозировать будущие требования к ресурсам. Такой подход поможет вовремя планировать масштабирование системы.

4. **Оптимизация.** Используем полученные данные для оптимизации ресурсов. Например, если обнаружим, что определенные серверы перегружены, в то время как другие недогружены, можно попытаться перераспределить нагрузку более эффективно.

5. **Планирование ресурсов.** Оцените, нужно ли увеличивать или уменьшать количество серверов в вашей инфраструктуре, основываясь на текущих и прогнозируемых нагрузках.

## Какие методы и инструменты вы используете для прогнозирования роста загрузки и потребностей в ресурсах?
<a id="Methods-and-Tools-for-Forecasting-Load-Growth"></a>
([наверх](#sections))

Некоторые из наиболее широко используемых методов и инструментов:

### Методы прогнозирования:

1. **Метод экстраполяции.** Этот метод основывается на анализе исторических данных о загрузке и ресурсах. Он предполагает, что будущие показатели будут следовать тем же тенденциям, что и прошлые. Экстраполяция может включать в себя использование методов регрессионного анализа, временных рядов и других статистических методов.

2. **Метод сценариев.** Этот метод предполагает создание различных сценариев для прогнозирования. Например, можно создать сценарии на основе ожидаемого роста клиентской базы, внедрения новых функций или изменений в алгоритмах обработки данных.

3. **Метод экспертной оценки.** Использование опыта и знаний экспертов в области, чтобы прогнозировать будущие потребности в ресурсах. Этот метод может быть особенно полезен при прогнозировании новых технологических трендов или изменений в бизнес-модели компании.

### Инструменты для прогнозирования:

1. **Системы мониторинга и аналитики.** Используйте специализированные инструменты мониторинга, такие как Prometheus, Nagios, Zabbix, Grafana и другие, для сбора, анализа и визуализации данных о производительности системы.

2. **Программное обеспечение для анализа данных.** Используйте инструменты анализа данных, такие как Python с библиотеками pandas, NumPy и scikit-learn, для проведения статистического анализа и построения прогнозных моделей.

3. **Системы управления облачными ресурсами.** Для организаций, использующих облачные ресурсы, такие как Amazon Web Services (AWS), Microsoft Azure или Google Cloud Platform, существуют специализированные инструменты и службы, позволяющие автоматизировать масштабирование инфраструктуры в соответствии с прогнозами нагрузки.

4. **Моделирование и симуляция.** Используйте программные инструменты для моделирования и симуляции, которые позволяют создавать виртуальные модели системы и тестировать различные сценарии нагрузки.

5. **Интегрированные платформы капасити менеджмента.** Существуют специализированные интегрированные платформы для управления капасити, которые объединяют в себе различные инструменты и методы прогнозирования, облегчая процесс принятия решений по управлению ресурсами.

## Какие факторы вы учитываете при прогнозировании капасити?
<a id="Factors-Considered-in-Capacity-Forecasting"></a>
([наверх](#sections))

При прогнозировании капасити, то есть определении необходимого уровня ресурсов для поддержки бизнес-процессов и ожидаемой нагрузки, необходимо учитывать ряд различных факторов. Ниже представлены основные из них:

1. **Исторические данные.** Анализ прошлой загрузки и использования ресурсов позволяет выявить сезонные колебания, тренды роста, а также определить типичные паттерны поведения системы.

2. **Бизнес-требования** Необходимо понять текущие и будущие бизнес-потребности компании, включая планы по расширению бизнеса, запуску новых продуктов или услуг, а также изменения в бизнес-модели.

3. **Требования пользователей.** Изучение поведения пользователей и их ожиданий от системы помогает оценить ожидаемую нагрузку и выявить ключевые моменты пиковой активности.

4. **Технологические изменения.** Планирование обновлений систем, внедрение новых технологий, а также изменения в архитектуре приложений могут существенно повлиять на потребность в ресурсах.

5. **Прогнозирование роста бизнеса.** Проекция роста клиентской базы, объема транзакций, объема данных и других параметров, связанных с бизнесом, помогает определить ожидаемый рост нагрузки на систему.

6. **Сезонные колебания.** В зависимости от типа бизнеса могут существовать сезонные колебания в активности, которые необходимо учитывать при прогнозировании капасити.

7. **Требования к производительности и отказоустойчивости.** Некоторые приложения могут иметь особенно высокие требования к производительности и отказоустойчивости, что также нужно учитывать при расчете капасити.

8. **Запасы и гибкость.** Важно учитывать не только текущие потребности, но и предусмотреть запасные мощности для покрытия неожиданных скачков нагрузки и обеспечения гибкости системы.

9. **Законодательные требования и стандарты безопасности.** Соблюдение законодательства и стандартов безопасности может потребовать дополнительных ресурсов для хранения и обработки данных.

10. **Сроки обновлений и масштабирования.** Планирование сроков обновлений и масштабирования системы важно для обеспечения бесперебойной работы и избежания простоев.


# Оптимизация использования ресурсов
<a id="Resource-Utilization-Optimization"></a>
([наверх](#sections))

## Какие стратегии вы предложите для оптимизации использования ресурсов?
<a id="Strategies-for-Resource-Optimization"></a>
([наверх](#sections))

Задача тут простая в формулировке и сложная в реализации. Есть сервер с фиксированным железом, есть поток запросов, есть SLO по латентности и ошибкам. Нужно так организовать работу приложения и архитектуры, чтобы при той же физической конфигурации выдерживать больше полезной нагрузки.

Формально можно считать, что для одного запроса есть ресурсный профиль
($T_{cpu_{req}}$, $B_{mem_{req}}$, $n_{io_{req}}$, $B_{io_{req}}$, $B_{net_{req}}$)
CPU время, память, операции ввода-вывода, объем данных на диске и по сети. Для сервера есть соответствующие потолки
($C_{cpu}$, $RAM$, $IOPS$, $BW_{disk}$, $BW_{net}$).

Тогда верхнюю оценку капасити удобно записывать так

$$
RPS_{max} \approx \min\left(
\frac{C_{cpu}}{T_{cpu_{req}}},
\frac{BW_{mem}}{B_{mem_{req}}},
\frac{IOPS}{n_{io_{req}}},
\frac{BW_{disk}}{B_{io_{req}}},
\frac{BW_{net}}{B_{net_{req}}}
\right) \cdot k_{safety}
$$

Стратегии оптимизации ресурсов сводятся к двум вещам,
уменьшить "цену" одного запроса по ключевым ресурсам и сделать поведение более ровным во времени, чтобы не убиваться в пики. Дальше по ресурсам и уровням системы.

**CPU оптимизация**

CPU ограничивает капасити в двух сценариях. Или запросы сами по себе тяжелые, или запросов много, а каждый содержит избыточное количество логики. В обоих случаях нас интересует величина $T_{cpu_{req}}$ - сколько миллисекунд CPU тратит на один запрос при типичной нагрузке.

Если разложить время на операции, получится что-то вроде

$$
T_{cpu_req} = \sum_i N_i \cdot t_i
$$

где $N_i$ количество раз, когда выполняется операция $i$, $t_i$ ее средняя стоимость. Стратегии очевидные, но на практике все упирается в измерения.

1. Алгоритмы и структуры данных.
   Переход с $O(n)$ на $O(log n)$ или $O(1)$ для горячих операций почти всегда дает больший эффект, чем любые микроскопические оптимизации. Типичный пример - замена линейного поиска по списку конфигураций на хеш-таблицу, пересчет агрегатов по окнам вместо полного скана, индекс на поле с фильтром.

2. Избавление от лишней работы.
   Часто запрос делает много необязательных действий. Слишком подробные логи в горячем пути, постоянная сериализация больших объектов, лишние запросы к базе. Если на уровне профайлера видно, что 30 процентов CPU уходит в логгер, то самый быстрый код - тот, который не выполняется.

3. Эффективные форматы данных.
   JSON с большими вложенными структурами стоит дорого. Если на критическом пути заменить его на protobuf или компактный бинарный формат, $T_{cpu_{req}}$ падает, а с ним растет $RPS_{max}$. Условный пример. Было 5 млн инструкций на запрос, стало 3 млн. Для сервера из прошлого раздела это меняет оценку

$$
RPS_{cpu}^{old} \approx \frac{112 \cdot 10^9}{5 \cdot 10^6} \approx 22\ 400
$$

$$
RPS_{cpu}^{new} \approx \frac{112 \cdot 10^9}{3 \cdot 10^6} \approx 37\ 300
$$

в реальности эффект меньше из-за других ограничений, но пропорция сохраняется.

4. Управление параллелизмом.
   Чем больше потоков без необходимости, тем выше стоимость синхронизации и переключений. Есть полезное правило. Максимальное число рабочих потоков для CPU bound сервиса выбирается около $N_{cores}$ или немного больше. Для IO bound может быть больше, но это проверяется нагрузочным тестом.

Из практики. Если у сервиса 16 ядер, а пул потоков 200, очень часто процессор тратит существенную долю времени на переключения, а не на полезную работу. После уменьшения пула до 32 потоков latency падает, а стабильный RPS растет.

**Работа с памятью**

Дешёвая памть заканчивается в тот момент, когда рабочий набор перестает помещаться в RAM, начинается конкуренция за кеши и система падает в диск. Оптимизация памяти касается трех вещей. Рабочий набор, размер живых объектов и поведение кешей.

Удобно мыслить через вероятность того, что данные окажутся "рядом" с CPU. Пусть $p_{mem}$ доля обращений, которые обслуживаются из RAM, $p_{disk}$ доля, которая уходит на диск. Тогда средняя латентность доступа

$$
L_{data} = p_{mem} \cdot L_{mem} + p_{disk} \cdot L_{disk}
$$

Если $L_{disk}$ на два порядка больше $L_{mem}$, то даже несколько процентов дисковых обращений заметно портят картину.

Стратегии оптимизации.

1. Ограничение рабочего набора.
   Вместо загрузки огромных коллекций в память стоит переходить на стриминг, курсоры, постраничную обработку. В ETL пайплайнах это переход с "прочитал все в список, обработал" на "читаю порциями по N элементов". При этом $B_{mem_{req}}$ для одного запроса падает, а вероятность $p_{disk}$ снижается.

2. Оптимизация представления данных.
   Крупные графы объектных моделей в JVM или .NET занимают в разы больше памяти, чем тот же набор данных в виде компактных struct или массивов примитивов. Уменьшение overhead на один объект очень быстро видно на heap.

Пример. Было 10 млн объектов по примерно 200 байт на каждый, всего 2 ГБ. После пересмотра схемы хранения и перехода на более плотный формат каждый объект стал занимать 80 байт. Рабочий набор сократился до 800 МБ, исчезли лишние сборки мусора, latency перестала прыгать.

3. Настройка кешей.
   Кеш, который занимает "все, что осталось" от памяти, иногда вреднее, чем отсутствие кеша. Обычно выбирают ограниченный размер, соответствующий доле RAM, и смотрят на хитрейт.

Если обозначить $hit_{rate}$ как вероятность попадания в кеш, $L_{backend}$ как латентность похода в базу или другой источник, а $L_{cache}$ как латентность кеша, то среднее время доступа

$$
L_{total} = hit_{rate} \cdot L_{cache} + (1 - hit_rate) \cdot L_{backend}
$$

Оптимизация ресурсов тут в том, чтобы при разумном расходе памяти сделать $hit_rate$ как можно выше на самых горячих ключах. Иногда выгодно ввести два уровня кеша. Малый in-process для самых часто используемых вещей и внешний, типа Redis, для остального.

**Диск и ввод-вывод**

Диск плохо масштабируется на мелкие хаотичные операции. Поэтому цель оптимизации - уменьшить количество $n_{io_{req}}$ и сделать размер $B_{io_{req}}$ более предсказуемым и последовательным.

1. Сокращение количества обращений.
   Частый паттерн. HTTP запрос вызывает репозиторий, тот делает N запросов к базе вместо одного, потом еще читает файлы конфигурации в runtime. На практике миграция с "много мелких SQL" на один запрос с join или подзапросом дает больший выигрыш, чем различия между SSD моделей.

Если было 10 случайных чтений на запрос при IOPS 10 000, то верхняя оценка

$$
RPS_{disk} \approx \frac{10\ 000}{10} = 1000
$$

После оптимизации до 3 чтений на запрос

$$
RPS_{disk} \approx \frac{10\ 000}{3} \approx 3300
$$

2. Переход к последовательному чтению и батчированию.
   Журналы, очереди, временные данные лучше писать большими порциями. В логгерах стоит убирать синхронные fsync на каждом сообщении. В ETL пайплайнах агрегировать батчи, чтобы уменьшить число операций записи.

3. Предвыборка и прогрев.
   Для запросов, где заранее понятен набор страниц или сегментов, полезно использовать prefetch. Это не только оптимизация скорости, но и стабилизация латентности. Если самые горячие таблицы и индексы подгружены в файловый кеш, вероятность провала в диск гораздо ниже.

**Сеть**

Сеть ограничивает капасити в двух аспектах. Ограниченный $BW_{net}$ и ограниченный $PPS_{max}$ по количеству пакетов. Оптимизация идет по двум направлениям. Поменять протокол или формат, уменьшив $B_{net_{req}}$, и изменить схему взаимодействия между сервисами.

1. Размер запросов.
   Грубая, но работающая стратегия. На критическом пути использовать более компактные форматы. Урезать payload до реально нужного, убрать лишние поля, сократить вложенность. Компрессия помогает, но требует CPU.

Если средний размер ответа сокращается с 50 КБ до 20 КБ, а линк тот же 1Gbit, то оценка

$$
RPS_{net}^{old} \approx \frac{125 \cdot 10^6}{50 \cdot 10^3} = 2500
$$

$$
RPS_{net}^{new} \approx \frac{125 \cdot 10^6}{20 \cdot 10^3} = 6250
$$

2. Уменьшение числа hops.
   Каждый лишний микросервис в цепочке запроса добавляет свою долю latency и трафика. Частая оптимизация. Часть данных подтягивается заранее в один сервис, чтобы клиент не дергал пять разных.

3. Устранение N+1.
   Когда один запрос к API вызывает десятки запросов к другим сервисам, и каждый тратит по несколько миллисекунд, суммарно нагрузка по сети растет нелинейно. Оптимизация на уровне "договориться о bulk интерфейсе" снижает и $B_{net_{req}}$, и $n_{req_{net}}$.

**Параллелизм, очереди и backpressure**

Даже при хорошем коде можно убить ресурсы неправильным управлением параллелизмом. Тут полезно вспомнить закон Литтла

$$
L = \lambda \cdot W
$$

где $L$ среднее количество запросов в системе, $λ$ входной поток, $W$ среднее время обработки. Если $λ$ приближается к потолку $RPS_{max}$, $W$ растет, а вместе с ним растут очереди и потребление ресурсов.

Стратегии на этом уровне.

1. Ограничение числа параллельных запросов.
   Вместо того чтобы принимать все входящие соединения подряд, сервис держит фиксированный пул рабочих задач. Остальные ожидают в очереди или получают отказ. Это предотвращает ситуацию, когда все ресурсы уходят на попытки обработать слишком большой поток, и в итоге проваливаются все.

2. Backpressure.
   Система должна уметь "сказать" потребителю, что сейчас ей нужно сбавить скорость. Это может быть HTTP 429, замедление ответов, временное снижение приоритета. Без такого механизма любое внешнее увеличение $λ$ приводит к каскаду ретраев и лавинообразному росту нагрузки на ресурсы.

3. Приоритеты.
   Иногда выгодно пожертвовать второстепенными запросами, чтобы сохранить SLO для критичных. Например, фоновая аналитика получает ограничение по RPS или запускается только при свободном CPU и диске.

**Кэширование и перераспределение работы**

Кэширование в контексте ресурсов - это обмен RAM и CPU высокой частоты на уменьшение нагрузки на медленные или дорогие части системы.

1. Вынос тяжелых вычислений из критического пути.
   Если часть логики можно предвычислить и положить в кеш, то $T_{cpu_{req}}$ и $n_{io_{req}}$ на основном пути уменьшаются. Например, пересчет пользовательских агрегатов ночью и хранение их в отдельной таблице с быстрым доступом вместо расчета "на лету" при каждом запросе.

2. Выбор уровня кеша.
   Иногда достаточно локального кеша в процессе, в других случаях лучше вынести его в отдельный сервис типа Redis. Тут важно учитывать, как это влияет на ресурсы. Локальный кеш расходует RAM на каждом инстансе, но не тратит сеть. Внешний кеш экономит RAM приложений, но требует трафика и отдельного сервера.

3. Инвалидизация.
   Неудачная стратегия инвалидизации превращает кеш в генератор шторма при обновлениях, когда масса запросов одновременно промахивается и идет в базу. Правильные TTL, ключи, логика обновления - это защита от всплесков по ресурсам.

**Архитектурные стратегии**

Есть оптимизации, которые не видно в профилировщике одного сервиса, но которые радикально меняют нагрузку на ресурсы.

1. Асинхронность и очереди.
   Все, что не критично к немедленному ответу пользователю, лучше вынести в асинхронный бэкграунд с очередями. Запись логов, тяжелые интеграции, нотификации. Это сглаживает пики нагрузки и позволяет планировать работу по ресурсам во времени.

2. Разделение чтения и записи.
   Чтение и запись обычно имеют разные профили нагрузки. Реплики базы под чтение разгружают мастер, который занимается записью. Для DWH расчет витрин и их чтение тоже часто разделяют по кластерам.

3. Приближенные вычисления.
   Для ряда задач точность не критична. Вместо тяжелых точных расчетов можно использовать сэмплинг, HyperLogLog, approximate distinct и так далее. База в этот момент перестает упираться в дисковые и CPU ресурсы на скане гигантских таблиц.

4. Денормализация и предагрегаты.
   Отношения в стиле "нормализовали все до упора" хорошо выглядят в учебниках, но плохо на ресурсах при реальных нагрузках. Предагрегированные таблицы, материализованные представления, частичная денормализация уменьшают количество join и объем данных, который нужно гонять по каждому запросу.

**Практический алгоритм оптимизации**

Чтобы этот раздел работал как часть базы знаний, удобно иметь простой сценарий действий.

Сначала измеряется текущий профиль под нагрузкой. Для типичного запроса можно оценить

* CPU время на уровне профайлера
* объем живых данных и поведение GC
* количество и профиль I/O в базу и файловую систему
* трафик по сети и число внешних запросов

Дальше строится примитивная модель вида

$$
RPS_{cpu}, RPS_{ram}, RPS_{disk}, RPS_{net}
$$

и выбирается минимальное, то есть текущий лимит. После этого цикл.

1. Оптимизировать самый узкий ресурс так, чтобы уменьшить его "цену" на запрос.
2. Перемерить профиль.
3. Обновить модель и найти новый лимит.
4. Повторять, пока системный лимит не упрется в физическую конфигурацию или в внешние системы, которые уже не твой сервер.

В итоге получается связка двух глав. Одна описывает, как физическая конфигурация задает верхние границы по ресурсам. Вторая показывает, как код и архитектура позволяют приблизиться к этим границам и потратить каждый гигагерц, гигабайт и мегабайт в секунду максимально осмысленно.


## Как вы будете реагировать на неэффективное использование ресурсов?
<a id="Responding-to-Inefficient-Resource-Use"></a>
([наверх](#sections))

Реагировать необходимо по трём фронтам. Стабилизация прямо сейчас. Диагностика и локализация причины. Исправление и защита от повторения.

Смысл "неэффективности" в том, что система тратит ресурс, но не превращает его в полезный throughput. Типовая картина, RPS не растет, латентность растет, CPU или диск в потолок, либо наоборот, CPU низкий, а очередь и таймауты растут из-за блокировок, внешних зависимостей, лимитов пулов.

**1) Зафиксировать симптом и выбрать метрику эффективности**

Сначала привязываю проблему к SLO. Что именно ухудшилось, p95, p99, ошибки, таймауты, рост очередей, деградация конкретного эндпоинта. Дальше выбираю "стоимость запроса" по главному ресурсу.

Если подозрение на CPU, то опорная величина это $T_{cpu_{req}}$, среднее CPU время на запрос. Тогда грубая граница такая

$$RPS_{cpu} \approx \frac{C_{cpu}}{T_{cpu_{req}}}$$

Если растет $T_{cpu_{req}}$, капасити падает даже при том же железе. Это сразу превращается в проверяемую гипотезу.

Если подозрение на сеть

$$RPS_{net} \approx \frac{BW_{net}}{B_{net_{req}}}$$

Если подозрение на диск

$$RPS_{disk} \approx \min\left(\frac{IOPS}{n_{io_{req}}}, \frac{BW_{disk}}{B_{io_{req}}}\right)$$

Эти оценки нужны не для точности, они помогают быстро понять, куда смотреть в первую очередь и что именно оптимизировать.

**2) Быстро стабилизировать, чтобы остановить лавину**

Пока идет поиск причины, нужно не дать системе уйти в штопор. Я делаю это так.

Ограничиваю входной поток. Rate limit, очереди, защита от ретраев, временное снижение функциональности для тяжелых сценариев. При необходимости включаю деградацию, например отдача данных из кеша, упрощенный ответ, отключение вторичных фич.

Включаю backpressure. Если сервис физически не успевает, он должен говорить об этом вовне. Иначе начинается петля, таймаут, ретрай, еще больше нагрузки, еще больше таймаутов.

Снимаю лишний фон. Тяжелые джобы, бэкапы, компакции, переиндексации, любой batch, который конкурирует за диск и CPU. Это особенно важно, когда упор в I/O.

Снижаю шум. Логи на горячем пути, debug трассировки, чрезмерные метрики. На пике это легко превращается в отдельный потребитель CPU и диска.

Если есть безопасный быстрый вариант, делаю масштабирование. Горизонтальное на stateless слое. Вертикальное реже, обычно дороже и медленнее. При этом я не считаю скейл лечением. Это способ выиграть время и сохранить SLO.

**3) Локализовать узкое место через очереди и закон Литтла**

Очень полезно смотреть на очереди в системе. Очередь запросов к пулу потоков, очередь соединений к базе, очередь на диск, очередь сообщений.

Закон Литтла дает понятный язык

$$L = \lambda \cdot W$$

Если растет $L$, значит растет $W$ при том же $\lambda$, либо вырос $\lambda$ при том же $W$. Дальше вопрос, что именно увеличило $W$. CPU, блокировки, ожидание сети, ожидание диска, лимиты пулов.

На практике я всегда пытаюсь найти конкретный "бутылочный горлышко" с измеримой метрикой. Например, рост времени ожидания соединения к Postgres, рост iowait, рост p99 на конкретном downstream, рост времени в GC, рост lock contention.

**4) Снять диагностические артефакты до того, как все рассосется**

Неэффективность часто плавающая, и после стабилизации исчезает. Поэтому на пике я собираю данные, которые потом невозможно получить.

Для CPU. Профили, flamegraph, pprof, async-profiler, perf, eBPF. Для JVM плюс thread dump. Для Python это cProfile, py-spy, scalene, плюс трассировки.

Для памяти. Heap dump, статистика аллокаций, пики RSS, признаки swap, рост времени GC. Важно отделять утечку от всплеска рабочего набора.

Для диска. iostat, latency по устройству, очередь, fsync, журналирование, slow log БД, wait events.

Для сети. Saturation интерфейсов, retransmits, рост p99 на gRPC, таймауты, увеличение payload.

Для баз данных. EXPLAIN ANALYZE, планы, статистика, блокировки, hot queries, N+1, рост временных файлов.

**5) Типовые причины неэффективности и как я на них реагирую**

CPU горит, RPS не растет. Часто это или рост $T_{cpu_{req}}$ из-за форматов и сериализации, или из-за аллокаций и GC, или из-за блокировок.

Реакция. Профиль CPU, найти топ функций. Проверить объекты и аллокации. Проверить синхронизацию и локи. Проверить размер пулов и контекстные переключения. После этого фиксы обычно такие, убрать лишнюю сериализацию, сменить формат, оптимизировать алгоритм, уменьшить аллокации, убрать логирование с формированием строк на горячем пути, переработать критическую секцию.

CPU низкий, латентность высокая. Часто упор во внешнюю зависимость или в лимиты пулов. Например, пул соединений к базе маленький, запросы ждут, CPU простаивает.

Реакция. Смотрю ожидание на семафорах и пулах. Смотрю трассы по downstream. Проверяю таймауты, ретраи, circuit breaker. Поднимаю лимиты аккуратно и измеряю, чтобы не убить БД.

Память растет и сервис начинает деградировать. Варианта два, утечка или рабочий набор перестал помещаться в RAM. В обоих случаях $p_{disk}$ растет, и система проваливается в I/O.

Реакция. Проверяю тренд heap, RSS, GC time. Смотрю, что именно растет по типам объектов. Если утечка, фикс владения объектами, кэши, ссылки, коллекции. Если рабочий набор, меняю модель обработки, стриминг, батчи, ограничение кеша, eviction, вынесение тяжелого в асинхронный слой.

Диск в потолок. Часто это много мелких операций и fsync, либо конкуренция фоновых процессов, либо плохие запросы к БД, которые создают временные файлы.

Реакция. Уменьшаю $n_{io_{req}}$ и делаю I/O более последовательным. Батчирование, буферизация логов, перенос временных файлов на быстрый диск, настройка checkpoints, пересмотр запросов и индексов.

Сеть упирается в throughput или pps. Симптом, потолок RPS на ровном месте, рост retransmits, рост latency межсервиса.

Реакция. Уменьшаю $B_{net_{req}}$, сжимаю payload, убираю лишние поля, ввожу bulk API, сокращаю hops, кэширую ближе к потребителю.

**6) Внести исправление так, чтобы оно не повторилось**

После локализации причины я делаю fix и добавляю защиту.

Если это код, то добавляю бенчмарк и регрессионный тест по профилю ресурса. Например, тест на верхнюю границу $T_{cpu_{req}}$ для критичного эндпоинта на фиксированном наборе данных.

Если это конфигурация, то фиксирую в IaC и добавляю алерт на дрейф. Пулы, лимиты, таймауты, ретраи, приоритеты.

Если это нагрузка, то добавляю ограничители. Rate limit по ключам, per-tenant квоты, ограничение тяжелых запросов, защита от N+1 и от "случайного full scan".

Если это внешняя зависимость, то усиливаю изоляцию. Circuit breaker, fallback, кеш, bulkheads.

И обязательно добавляю наблюдаемость на стоимость запроса по ресурсам. Для CPU это распределение $T_{cpu_{req}}$ или proxy через time in handler и профили по выборке. Для БД это p95 и p99 по типам запросов, lock waits, temp files. Для сети это размер ответов и число downstream вызовов.

## Какие методы вы используете для определения узких мест и повышения эффективности использования капасити?
<a id="Methods-for-Identifying-Bottlenecks-and-Improving-Capacity-Utilization"></a>
([наверх](#sections))

Узкое место - это ресурс, который первым выходит в насыщение, и именно он определяет потолок $RPS_{max}$. Поэтому я действую циклом, измерил, локализовал, подтвердил, оптимизировал, снова измерил. Важный момент, я всегда разделяю проблему на два вопроса. Где мы теряем время, и где мы жжем ресурс без полезного результата.

**1) Как я определяю узкое место**

Я начинаю с картины на уровне сервиса и хоста. На сервисе беру latency распределениями p50 p95 p99, throughput, ошибки, длины очередей и отказы. На хосте смотрю CPU utilization и iowait, память и признаки давления, диск по latency и очереди, сеть по пропускной способности и retransmits. Дальше ищу сигнатуру.

Если CPU близко к потолку и latency растет с ростом RPS, то почти наверняка лимит по CPU. Если CPU низкий, а latency высокий и растет очередь, то обычно это ожидание, сеть, диск, блокировки, лимиты пулов, downstream.

Очень помогает закон Литтла. Я постоянно проверяю согласованность очередей и времени обработки.

$$L = \lambda \cdot W$$

Если $L$ растет, значит растет $W$ при том же $\lambda$, или вырос $\lambda$ при том же $W$. Это быстро отсекает ложные гипотезы, например "мало CPU", когда на самом деле выросло ожидание соединения к базе.

Еще одна полезная штука, оценка потолков по ресурсам на уровне порядка величины. Она нужна, чтобы сразу понять, какой ресурс вообще способен быть лимитом при текущем профиле запроса.

$$RPS_{cpu} \approx \frac{C_{cpu}}{T_{cpu_{req}}}$$

$$RPS_{disk} \approx \min\left(\frac{IOPS}{n_{io_{req}}}, \frac{BW_{disk}}{B_{io_{req}}}\right)$$

$$RPS_{net} \approx \frac{BW_{net}}{B_{net_{req}}}$$

Если измеренный RPS близок к одной из этих границ, я почти наверняка нашел текущий лимит. Если нет, значит есть скрытая причина, например блокировки, контеншн, GC, маленькие пулы, неудачные ретраи.

**2) Как я локализую место, где тратится время**

Самый быстрый путь - разложить latency на составные части через трассировку. Tracing дает ответ, где именно живет $W$, в приложении, в БД, в сети, в очереди, в ожидании блокировки, в ожидании пула. Я обычно стремлюсь получить разложение вида, обработчик, ожидание пула, downstream calls, сериализация, доступ к диску, прочее. После этого узкое место становится конкретным участком.

Если упор в CPU, я делаю профилирование, flamegraph, pprof, async-profiler, perf или eBPF. Меня интересует, что съедает $T_{cpu_{req}}$. Часто это сериализация, регулярки, парсинг, аллокации, крипта, логирование, сортировки, конвертации типов.

Если подозрение на память, я смотрю на тренд heap RSS, GC time, частоту сборок, аллокации. Дальше heap dump и топ типов объектов. Это отвечает на вопрос, вырос рабочий набор или есть утечка, и что именно раздувает память.

Если упор в диск, я смотрю disk latency и queue depth. Для БД дополнительно slow queries, wait events, temp files, checkpoints. Дальше нахожу, какие операции дают высокий $n_{io_{req}}$ или большой $B_{io_{req}}$, и почему это стало происходить, например ушли индексы, поменялся план, добавился fsync на горячем пути.

Если упор в сеть, я смотрю средний размер запросов и ответов, число downstream вызовов на один входной запрос, retransmits, p99 межсервисных вызовов. Это быстро показывает, лимит по $BW_{net}$ или по pps, и где именно растет $B_{net_{req}}$ или количество hops.

**3) Как я повышаю эффективность использования капасити**

Тут я действую от текущего лимита. Цель, уменьшить стоимость одного запроса по узкому ресурсу и убрать вариативность, из-за которой p99 улетает раньше среднего.

Если лимит по CPU, я уменьшаю $T_{cpu_{req}}$. Самые частые рабочие ходы, убрать лишнюю работу на горячем пути, сократить сериализацию и конвертации, уменьшить аллокации, пересмотреть алгоритм, поменять формат данных, оптимизировать критические секции и снизить контеншн. Часто капасити растет просто от того, что CPU перестает тратить время на "служебное".

Если лимит по памяти, я уменьшаю рабочий набор и стабилизирую GC. Делается через стриминг и батчи, ограничение кешей, более плотные структуры данных, отказ от хранения больших промежуточных объектов, правильная политика eviction. Эффект обычно двойной, меньше память, меньше GC, меньше хвостов по p99.

Если лимит по диску, я снижаю $n_{io_{req}}$ и делаю I/O более крупным и последовательным. Батчирование записи, буферизация логов, перенос тяжелых временных файлов на быстрый диск, оптимизация запросов и индексов, снижение количества round-trip к БД, устранение N+1. Почти всегда помогает убрать конкуренцию фоновых задач за тот же диск.

Если лимит по сети, я уменьшаю $B_{net_{req}}$ и число hops. Урезаю payload до нужных полей, ввожу bulk API, кэширую ближе к потребителю, сокращаю межсервисные чаты, пересматриваю ретраи и таймауты, чтобы не создавать шторм. С компрессией аккуратно, она меняет ограничение с сети на CPU.

Отдельный класс оптимизаций - управление параллелизмом. Если система перегружается, она начинает тратить ресурс на очереди, ретраи, таймауты и контекстные переключения, и это выглядит как "неэффективность". Поэтому я почти всегда ввожу явные лимиты concurrency, backpressure, приоритеты, bulkheads. Это повышает полезную утилизацию ресурсов и держит хвосты latency.

**4) Как я подтверждаю, что стало лучше, и защищаюсь от регресса**

Любая оптимизация должна быть подтверждена измерением на одинаковом профиле нагрузки. Я всегда сравниваю кривые "RPS - p95 p99", стоимость запроса по узкому ресурсу, например $T_{cpu_{req}}$ или $n_{io_{req}}$, и headroom по ключевому ресурсу. Если после фикса потолок сдвинулся, узкое место обычно переезжает на другой ресурс, и цикл повторяется.

Чтобы это не ломалось снова, я добавляю метрики стоимости запроса и алерты на дрейф, например рост $T_{cpu_{req}}$, рост числа downstream вызовов на запрос, рост объема ответа, рост wait time по пулу соединений, рост temp files в БД. Это позволяет ловить деградации до того, как они превращаются в инцидент.


# Управление проектами Capacity
<a id="Capacity-Project-Management"></a>
([наверх](#sections))

## Как вы будете управлять портфелем проектов Capacity?
<a id="Managing-the-Capacity-Project-Portfolio"></a>
([наверх](#sections))

Управление портфелем проектов Capacity я бы строил как продуктовую функцию с четкими правилами входа, прозрачной приоритизацией и ограничением одновременной работы. Смысл портфеля в том, чтобы в каждый момент времени делать небольшой набор инициатив, которые с максимальной вероятностью повышают headroom и снижают стоимость роста, при этом не съедая всю команду и не ломая основной delivery.

Сначала я фиксирую границы портфеля. Что мы называем Capacity проектом, какие системы входят, какой горизонт планирования, какие целевые уровни headroom и какие "красные зоны", где нельзя уходить ниже. Дальше завожу единый backlog инициатив Capacity, он общий для команд и платформ, чтобы не было параллельных решений одной и той же проблемы в разных углах.

У инициативы должен быть стандартный "паспорт". Короткое описание, затрагиваемые компоненты, предполагаемый эффект на headroom и стоимость, оценка усилий, зависимости, способ проверки результата, срок полезности, например временный костыль или долгосрочное изменение. Это дает одинаковый язык для обсуждения с продуктом, разработкой, SRE и безопасностью.

Приоритизацию я делаю не по громкости запроса, а по ожидаемому эффекту на ограничивающий ресурс и по цене промедления. Удобно использовать простую скоринговую модель, чтобы спорить о предположениях, а не о вкусах. Например так

$$Score = \frac{Impact \cdot Urgency \cdot Confidence}{Effort}$$

Тут $Impact$ это ожидаемый прирост headroom или снижение потребления ключевого ресурса на критичных путях, $Urgency$ это близость к насыщению или к бизнес-пику, $Confidence$ это качество данных и ясность механики, $Effort$ это календарная длительность с учетом занятых людей и зависимостей. Формула нужна как дисциплина, итоговое решение все равно принимает руководящий контур, но обсуждение становится конструктивным.

Дальше я разделяю портфель на 2-3 потока, чтобы не смешивать срочные ограничения и стратегические улучшения. Обычно есть поток срочного освобождения headroom, поток системных улучшений, поток подготовительных работ под будущий рост, например миграции, изменение схем данных, рефакторинг контуров масштабирования. На уровне портфеля я ограничиваю WIP, если команда держит слишком много активных инициатив, эффект почти всегда хуже, сроки расползаются, контекст шатает.

Планирование делаю волнами. Квартальная рамка задает приоритеты и бюджет людей, месячная корректирует состав активных инициатив, недельная синхронизация снимает блокеры и пересобирает очередность, если реальная нагрузка или планы бизнеса поменялись. При этом у каждого проекта есть владелец, который отвечает за результат, и есть согласованный набор стейкхолдеров, которые обязаны быть в контуре, например владельцы зависимых сервисов, DBA, сеть, безопасность.

Важная часть портфельного управления это зависимости и "временные окна". Capacity инициативы часто требуют выката, миграций, заморозок или координации нескольких команд. Поэтому я заранее бронирую окна изменений и свожу проекты так, чтобы они не конфликтовали за одни и те же ресурсы, например диск и сеть на кластере, или одних и тех же экспертов. Это сильно снижает скрытые простои, когда все ждут одного человека или одного изменения в платформе.

И последнее. Портфель Capacity нельзя вести как бесконечный список хотелок. Я всегда держу правило завершения. Проект считается закрытым только тогда, когда эффект подтвержден в продуктивном контуре и зафиксировано, что именно изменилось в возможности системы расти. Это защищает от ситуации, когда мы тратим месяцы, а headroom на деле не появляется, потому что уперлись в другой слой или сделали оптимизацию не на критическом пути.


## Какие ключевые метрики и инструменты вы будете использовать для отслеживания прогресса проектов?
<a id="Key-Metrics-and-Tools-for-Tracking-Project-Progress"></a>
([наверх](#sections))

Для отслеживания прогресса Capacity проектов я использую три слоя метрик. Метрики результата, метрики эффективности, метрики поставки. Их важно вести вместе, потому что можно "закрывать задачи" без прироста headroom, или получить прирост ценой деградации SLO.

**1) Метрики результата, то есть стало ли реально больше капасити**

Базовая метрика, headroom на целевом SLO. Я фиксирую целевую нагрузку, например $p95$ и $p99$ по latency, долю ошибок, и смотрю, какой максимальный устойчивый RPS система держит в этом режиме. Дальше считаю запас

$$Headroom = \frac{Capacity - Load}{Capacity}$$

Где $Capacity$ это устойчивый потолок при целевом SLO, $Load$ это текущая или плановая нагрузка.

На практике я храню две версии этой картины. Synthetic cap через нагрузочное тестирование и production cap по реальным метрикам в пиковых окнах. Это дает честную проверку, проект сдвинул потолок или просто улучшил поведение на стенде.

Что именно смотрю в дашборде результата

* кривые RPS - $p95$ и RPS - $p99$ для ключевых эндпоинтов, точка излома важнее среднего
* error budget и burn rate по SLO, чтобы не получить капасити через рост ошибок
* время в очередях и доля таймаутов, как ранний признак, что потолок близко

Инструменты
Grafana дашборды, Prometheus метрики, SLO панели, alerting. Для трассировки OpenTelemetry плюс Jaeger или Tempo, чтобы подтверждать, что изменения реально сократили критический путь.

**2) Метрики эффективности, то есть сколько ресурса стоит единица полезной работы**

Это про стоимость запроса. Для каждого критичного пути я фиксирую несколько устойчивых прокси, которые легко сравнивать до и после релиза

* $CPU_{ms}/req$ или доля CPU на полезной работе, отдельно user и system, отдельно iowait
* $DB\ calls/req$ и время на downstream, чтобы ловить N+1 и лишние hops
* $Bytes_{net}/req$ и средний размер ответа, чтобы видеть рост сетевой цены
* $IOPS/req$ и $Bytes_{io}/req$ для сервисов, где диск может стать лимитом
* cache hit rate и доля обращений в медленный источник, когда кеширование часть решения

Если проект обещает экономию, я требую, чтобы хотя бы один из этих показателей улучшился в ожидаемую сторону, и чтобы это коррелировало с ростом $Capacity$.

Инструменты

* профилирование pprof, async-profiler, perf, eBPF для подтверждения $CPU_{ms}/{req}$ на горячем пути
* метрики БД, pg_stat_statements и wait events для Postgres, system.query_log и системные таблицы для ClickHouse
* логи и трассы, чтобы связать рост стоимости запроса с конкретным типом трафика или кодовым путем

**3) Метрики поставки, то есть двигаемся ли мы к внедрению без самообмана**

Capacity проекты часто ломаются на "готово в коде, не в проде". Поэтому я измеряю прогресс как способность безопасно выкатывать изменения и получать подтвержденный эффект

* доля задач, дошедших до продакшена, а не только merged
* lead time от начала работы до прод релиза
* частота релизов и rollback rate, это показатель качества изменений
* доля автоматизированных проверок, тестов производительности, регрессов по SLO

Инструменты
Jira для статуса и зависимостей, GitHub или GitLab для контроля релизов, CI/CD для воспроизводимых прогонов, окружения для нагрузочного теста. Для тестов k6, Gatling, JMeter, Locust, плюс фиксированные сценарии с одинаковыми датасетами. Результаты тестов складываю в отдельный дашборд, чтобы видеть тренд, а не один удачный прогон.

**4) Как это оформляю, чтобы прогресс был прозрачным**

На каждый Capacity проект я делаю один живой дашборд, который содержит baseline, текущую версию и target. И две контрольные точки. Первая, достигли ли мы ожидаемой экономии на стоимость запроса. Вторая, сдвинули ли мы $Capacity$ при целевом SLO в прод контуре или хотя бы в репрезентативном синтетическом тесте. Это позволяет руководить портфелем по факту прироста капасити, а не по количеству закрытых задач.

## Как вы будете управлять рисками, связанными с проектами Capacity?
<a id="Managing-Risks-Associated-with-Capacity-Projects"></a>
([наверх](#sections))

Управление рисками в Capacity проектах я строю так, чтобы риск был видимым, измеримым и управляемым еще до того, как мы начали писать код. У таких проектов типовая проблема, эффект сложнее предсказать, чем сроки, и цена ошибки часто выражается в инциденте или деградации SLO.

Сначала я ввожу единый реестр рисков на портфель и на каждый проект. Риск описывается не общими словами, а как сценарий, что именно может пойти не так, какой будет ущерб, по каким ранним сигналам это заметим, кто владелец и какой план смягчения. Для приоритизации удобно скорить риск как произведение вероятности и ущерба, например $RiskScore = p \cdot I$. Это не математика ради математики, это дисциплина, чтобы не терять время на мелочи, когда есть риск высокого класса.

Дальше я работаю тремя контурами. Предотвращение, раннее обнаружение, безопасное восстановление.

Предотвращение начинается с проверки гипотезы. Самый частый риск Capacity проектов, мы оптимизируем не тот лимит. Поэтому до старта я требую baseline и явное подтверждение, что текущий потолок реально упирается в конкретный ресурс и на конкретном критическом пути. Если подтверждения нет, проект либо режется на короткий discovery этап, либо откладывается. Второй частый риск, недооценка зависимости. Capacity изменения почти всегда цепляют несколько компонентов, поэтому я фиксирую карту зависимостей и "окна изменений" заранее, иначе проект превращается в ожидание чужих команд.

Риски корректности и безопасности я считаю обязательными. Оптимизация, которая ускоряет, но портит данные или обходится без требований комплаенса, неприемлема. Поэтому на старте фиксируется, какие инварианты нельзя нарушить, например согласованность данных, идемпотентность, ограничения по доступам, требования по логированию. Под это делаются проверки и план отката.

Раннее обнаружение, это контрольные точки и триггеры. Я заранее определяю, какие сигналы считаются красными. Рост p99 на ключевом endpoint, рост ошибок, ухудшение времени ожидания по пулам, рост таймаутов downstream, рост очередей. При срабатывании триггера проект уходит в режим блокировки релиза или отката, без обсуждений в стиле "может рассосется". Чтобы это работало, у проекта есть заранее подготовленные дашборды и алерты, иначе триггеры существуют только на бумаге.

Безопасное восстановление, это техника выката. Для Capacity проектов я почти всегда использую staged rollout. Фича флаги, canary на малую долю трафика, постепенное увеличение, возможность мгновенного rollback, ограничение blast radius. Если изменение затрагивает хранилища или схемы, я предпочитаю обратимо совместимые миграции и двухфазный подход, сначала расширение, потом переключение, потом уборка. Там, где возможно, полезен shadow traffic или параллельный расчет, чтобы сравнить результаты без влияния на пользователя.

Отдельно управляю риском "улучшили одно, сломали другое". Это классика, оптимизация сдвигает узкое место и проявляет новый лимит, либо повышает нагрузку на соседнюю систему. Здесь помогает обязательная проверка эффекта на весь контур. Если проект ускорил сервис, но при этом увеличил нагрузку на базу и приблизил ее к насыщению, это считается незавершенным риском, и проект не закрывается, пока не будет понятен новый потолок и план, что с ним делать.

Риски сроков и людей я уменьшаю организационно. Ограничение WIP, явное резервирование экспертов, фиксированный календарь изменений, регулярный риск-ревью на портфельном уровне. Если критичный риск не удается снизить, я предпочитаю дробление на этапы с быстрым выходом в прод и проверкой, а не большую "переделку всего", которая живет в ветке месяцами.

И последнее, у каждого Capacity проекта должен быть план B. Если оптимизация не успевает или не дает ожидаемого эффекта, должна существовать согласованная стратегия защиты SLO, например временный rate limit, деградация тяжелых функций, перенос части нагрузки, дополнительное масштабирование как временная мера. Это не заменяет проект, это страховка, чтобы бизнес пик не превратился в аварию.


# Проектирование процессов
<a id="Process-Design"></a>
([наверх](#sections))

## С чего вы начнёте оптимизацию процессов Capacity Management?
<a id="Starting-Point-for-Capacity-Management-Process-Optimization"></a>
([наверх](#sections))

Начну с того, чтобы превратить Capacity Management из набора разрозненных реакций на инциденты в управляемый процесс с понятными входами, выходами и ритмом. Самый быстрый путь, зафиксировать текущую базу и договориться о правилах, по которым мы принимаем решения, иначе оптимизация процессов превратится в спор мнений.

Сначала я соберу карту системы и нагрузки в терминах бизнеса. Какие пользовательские сценарии и сервисы действительно критичны, какие окна пиков, какая сезонность, какие планы роста и где уже сейчас есть напряжение. Это нужно, чтобы Capacity не оптимизировался вслепую и чтобы команда понимала, какой именно рост мы обязаны выдержать, например х2 за квартал или просто пережить высокий сезон без деградаций.

Дальше фиксирую "baseline" текущего состояния. Какой реальный headroom на проде на пиковых окнах, где точка излома по латентности, какие лимиты по ресурсам чаще всего срабатывают, какие типы работ съедают ресурсы фоном, например бэкапы, компакции, отчеты. На этом шаге я делаю простую модель емкости. Не детальную симуляцию, а модель порядка величины, которая отвечает на вопрос, чем мы упремся при росте. Для процесса важно, чтобы модель была воспроизводимой и обновлялась регулярно, иначе она быстро становится ложной.

Параллельно выстраиваю единый контур ответственности. Кто владелец Capacity по доменам, кто принимает решения по инфраструктуре, кто по приложениям, кто по данным и БД. У Capacity проектов обычно много зависимостей, поэтому без явного "ownership" и правил эскалации процессы начинают буксовать.

Следующий шаг, формализую входы в процесс и минимальную документацию. Любая инициатива или запрос на Capacity должен приносить три вещи. Текущее ограничение и подтверждение, где оно проявляется. Ожидаемый эффект, например рост headroom или снижение стоимости запроса. План проверки в проде, иначе мы не поймем, был ли результат. Это резко уменьшает количество "проектов ради проектов".

Дальше ставлю ритм управления. Раз в неделю короткий triage, что болит прямо сейчас, что блокирует рост, что можно закрыть малыми усилиями. Раз в месяц пересмотр прогноза нагрузки и обновление capacity модели, чтобы портфель не ехал по инерции. Раз в квартал согласование целей и бюджета ресурсов вместе с продуктом и платформой, чтобы Capacity не жил отдельно от бизнеса.

И только после этого перехожу к оптимизации самих процедур, например как мы запускаем нагрузочные тесты, как планируем окна изменений, как проводим постмортемы capacity-инцидентов, как переиспользуем результаты профилирования. Если начинать с этого сразу, получится красивый набор регламентов без привязки к реальным ограничениям.

Итог первой фазы, за короткий срок появляется прозрачная картина, где потолок, насколько мы близко к нему, кто отвечает за устранение узких мест и как быстро мы переводим гипотезы в проверенный прирост headroom. Это и есть фундамент, на котором уже имеет смысл "докручивать" процессы дальше.

## Как вы будете измерять успех вашей стратегии и процессов Capacity Management?
<a id="Measuring-the-Success-of-Your-Capacity-Management-Strategy-and-Processes"></a>
([наверх](#sections))

Успех стратегии и процессов Capacity Management я измеряю не количеством сделанных инициатив, а тем, насколько предсказуемо система переживает рост нагрузки и насколько меньше стало аварийной работы вокруг капасити. Поэтому метрики делю на результат для бизнеса, качество прогнозирования, качество исполнения процесса и экономику.

**1) Результат, то есть стало ли безопаснее и больше headroom**

Ключевой показатель - запас до потолка на целевом SLO в проде. Его удобно считать так

$$Headroom = \frac{Capacity - Load}{Capacity}$$

Успех, когда headroom по критичным сценариям стабильно выше оговоренного порога, например 25-30% в пиковые окна, и не проседает при прогнозируемых всплесках.

Параллельно смотрю здоровье SLO через error budget и burn rate. Мне важно, чтобы прирост капасити не покупался деградацией качества. Если после оптимизаций burn rate на пиках перестал улетать, это прямой сигнал, что процессы работают.

Еще один практичный индикатор, снижение доли аварийных действий ради капасити. Меньше экстренных скейлов, меньше ручных ограничений трафика, меньше ночных переключений, потому что планирование стало попадать в реальность.

**2) Точность прогнозирования и управляемость планов роста**

Capacity Management ценен тем, что снижает сюрпризы. Поэтому я меряю точность прогнозов нагрузки и запасов.

Обычно использую простую ошибку прогноза, например MAPE

$$MAPE = \frac{1}{n}\sum_{i=1}^{n}\left|\frac{Forecast_i - Actual_i}{Actual_i}\right|$$

Успех, когда ошибка прогнозов по ключевым метрикам снижается, а решения по закупке ресурсов и инициативам принимаются заранее, а не в режиме пожара.

Отдельно смотрю дрейф модели. Если прогноз капасити регулярно промахивается из-за изменения профиля запросов, значит процесс должен раньше ловить изменения стоимости запроса и обновлять допущения.

**3) Качество исполнения процесса, то есть мы реально доводим до подтвержденного эффекта**

Тут мне важны метрики, которые защищают от самообмана.

Доля инициатив, у которых эффект подтвержден в проде и зафиксирован числом, а не словами. Хороший целевой уровень, чтобы большинство проектов закрывались только после измеримого сдвига, иначе портфель превращается в бесконечный рефакторинг.

Lead time от появления сигнала риска до выкатанного изменения. Если он падает, значит процесс стал меньше буксовать на согласованиях и зависимостях.

Доля незапланированных работ по капасити. Когда процессы зрелые, доля emergency задач уменьшается, а основная работа уходит в плановый контур.

Процент изменений, которые вызывают регресс по SLO или откаты. Если после оптимизаций вырос rollback rate, значит управление рисками и выкаты требуют усиления.

**4) Экономика, то есть сколько стоит единица капасити**

Capacity без экономики быстро превращается в бесконтрольный рост инфраструктуры. Поэтому считаю стоимость полезной нагрузки.

Например стоимость единицы throughput

$$CostPerThroughput = \frac{InfraCost}{Throughput}$$

Или более прикладной вариант для конкретного сервиса, стоимость 1k RPS или стоимость обработки одного события.

Успех, когда при росте нагрузки стоимость растет медленнее, чем нагрузка, и когда прирост капасити достигается оптимизациями, а не только добавлением железа.

**5) Как фиксирую успех в динамике**

Я задаю базовую линию до внедрения процесса и сравниваю тренды поквартально и помесячно. В отчетности оставляю небольшой набор показателей, headroom на SLO, burn rate, точность прогнозов, доля подтвержденных эффектов, доля emergency работ, стоимость единицы нагрузки. Если эти метрики улучшаются одновременно, значит стратегия и процессы Capacity Management реально работают.

## Как вы будете использовать SQL или Python для анализа данных в области Capacity Management?
<a id="Using-SQL-or-Python-for-Data-Analysis-in-Capacity-Management"></a>
([наверх](#sections))

Я использую SQL и Python как два дополняющих инструмента. SQL быстро дает честную картину по фактам в логах и метриках. Python нужен там, где начинается моделирование, прогнозирование и эксперименты "что будет, если".

**Как использую SQL в Capacity Management**

SQL я применяю, чтобы разложить нагрузку по компонентам и посчитать стоимость единицы работы. Обычно это делается на данных из query log базы, метрик приложения, reverse proxy, трассировки.

Первый шаг, получить профиль нагрузки по времени и по "ключам", endpoint, tenant, тип запроса, класс джобы. Простейшая основа, RPS и хвосты latency.

```sql
-- RPS и p95 по эндпоинтам по минутам
SELECT
  date_trunc('minute', ts) AS t,
  endpoint,
  count(*) AS req,
  percentile_cont(0.95) WITHIN GROUP (ORDER BY latency_ms) AS p95_ms
FROM api_requests
WHERE ts >= now() - interval '7 days'
GROUP BY 1, 2;
```

Дальше я ищу, где сгорает капасити, через "стоимость запроса". Для CPU это $CPU_{ms}/req$, для базы это число обращений, время в БД, объем прочитанных строк, для сети это $Bytes_{net}/req$. В SQL удобно сразу считать взвешенные показатели.

```sql
-- Средняя и p95 стоимость по CPU на один запрос
SELECT
  endpoint,
  count(*) AS req,
  avg(cpu_ms) AS cpu_ms_avg,
  percentile_cont(0.95) WITHIN GROUP (ORDER BY cpu_ms) AS cpu_ms_p95
FROM api_requests
WHERE ts >= now() - interval '24 hours'
GROUP BY 1
ORDER BY cpu_ms_p95 DESC
LIMIT 20;
```

Третий класс задач в SQL, найти "виновников" по ресурсам, top-N по нагрузке. Например, самые тяжелые SQL-запросы в Postgres через pg_stat_statements или в ClickHouse через system.query_log, либо джобы, которые создают пики I/O.

```sql
-- Top запросов по суммарному времени в БД
SELECT
  queryid,
  calls,
  total_time_ms,
  mean_time_ms,
  rows
FROM pg_stat_statements
ORDER BY total_time_ms DESC
LIMIT 20;
```

Четвертый класс задач, построение кривой насыщения. Я группирую интервалы по уровню нагрузки и смотрю, как ведет себя p95 и ошибки. Это помогает увидеть точку излома и отделить CPU-лимит от ожиданий.

```sql
-- Связь нагрузки и p95, группировка по "корзинам" RPS
WITH per_min AS (
  SELECT
    date_trunc('minute', ts) AS t,
    count(*)/60.0 AS rps,
    percentile_cont(0.95) WITHIN GROUP (ORDER BY latency_ms) AS p95_ms
  FROM api_requests
  WHERE ts >= now() - interval '3 days'
  GROUP BY 1
)
SELECT
  width_bucket(rps, 0, 5000, 20) AS rps_bucket,
  avg(rps) AS rps_avg,
  avg(p95_ms) AS p95_ms_avg
FROM per_min
GROUP BY 1
ORDER BY 1;
```

В итоге SQL дает базу для решений. Где именно тратится ресурс, как меняется стоимость запроса, какие сценарии формируют пики, какие конкретные запросы или endpoint упираются первыми.

**Как использую Python**

Python подключаю, когда нужно сделать то, что в SQL неудобно или слишком громоздко.

1. Прогноз нагрузки и сезонности. Беру таймсерии RPS, активных пользователей, объемов данных, добавляю календарные признаки, строю прогноз и доверительный интервал. Это превращает Capacity из реакции на "вчера было плохо" в планирование.

2. Детект аномалий и сдвигов. Например, резко изменился профиль трафика и вырос $CPU_{ms}/req$. Я ищу change point и связываю его с релизами, фичами, маркетинговыми кампаниями.

3. Моделирование "что будет, если". Сценарии роста, изменение лимитов пулов, включение компрессии, перенос части нагрузки в кеш. Тут Python удобен тем, что можно быстро прогнать симуляцию на исторических данных.

Пример мини-пайплайна, который связывает рост p95 с ростом стоимости запроса по CPU и помогает проверить гипотезу про деградацию кода после релиза.

```python
import pandas as pd

df = pd.read_parquet("api_minute_agg.parquet")  # t, rps, p95_ms, cpu_ms_avg, errors
df = df.sort_values("t")

# простая проверка, что p95 растет вместе с cpu_ms_avg при сопоставимом rps
corr = df[["p95_ms", "cpu_ms_avg", "rps"]].corr()
print(corr)

# поиск подозрительных минут, когда p95 вырос сильнее, чем обычно при данном rps
q = df.query("rps > 0")
baseline = q.groupby(pd.qcut(q["rps"], 20))["p95_ms"].median()
q["bucket"] = pd.qcut(q["rps"], 20)
q["p95_baseline"] = q["bucket"].map(baseline)
suspect = q[q["p95_ms"] > 1.5 * q["p95_baseline"]]
print(suspect[["t", "rps", "p95_ms", "cpu_ms_avg", "errors"]].tail(20))
```

Смысл такой. SQL отвечает на вопрос "что происходит и где". Python отвечает на вопрос "почему, что будет дальше и какой эффект дадут варианты".

## Какие типы аналитических отчетов и метрик вы будете разрабатывать для мониторинга капасити?
<a id="Types-of-Analytical-Reports-and-Metrics-for-Capacity-Monitoring"></a>
([наверх](#sections))

Я бы делал отчеты так, чтобы они отвечали на три практических вопроса. Сколько запаса есть сейчас. Где именно мы теряем капасити. Что будет, если нагрузка вырастет, и что нужно сделать заранее. Поэтому набор отчетов и метрик строю вокруг headroom, стоимости запроса, узких мест и прогноза.

Первый тип отчета, "Capacity overview" по критичным сервисам и сценариям. Он живет как дашборд и как еженедельный срез. В нем для каждого сервиса фиксируется текущая нагрузка, текущий cap на целевом SLO и запас, то есть headroom. Headroom считаю формулой

$$Headroom = \frac{Capacity - Load}{Capacity}$$

К нему обязательно добавляю burn rate по SLO и долю ошибок, чтобы рост капасити не был достигнут деградацией качества. Это главный управленческий отчет, по нему видно, кто в красной зоне и куда идти дальше.

Второй тип, "Saturation и точка излома". Это отчет, который показывает кривые RPS - p95 и RPS - p99, отдельно по ключевым эндпоинтам. Важен не средний уровень, а место, где начинается резкий рост хвостов. Я использую его для принятия решений, упираемся ли мы в ресурс, или в контеншн, или в downstream. В этом же отчете показываю очереди, длину пула, время ожидания, долю таймаутов. Он хорошо работает как мост между SRE и разработкой.

Третий тип, "Unit cost", стоимость единицы работы. Это набор метрик, которые показывают, сколько ресурсов потребляет один запрос или одно событие, и как это меняется со временем. На уровне приложений это $CPU_{ms}/req$, число downstream вызовов на запрос, средний размер ответа, аллокации или proxy через GC time. На уровне данных это $DB\ time/req$, количество SQL на запрос, строки или байты, обработанные на запрос, временные файлы. На уровне сети это $Bytes_{net}/req$ и PPS proxy через число вызовов и размер. Этот отчет нужен для раннего обнаружения регресса, когда новый релиз медленно увеличивает цену запроса, и headroom тает без роста трафика.

Четвертый тип, "Top drivers", что именно съедает ресурсы. Это ранжированные отчеты, топ эндпоинтов по p99, топ запросов к базе по total time, top tenants по нагрузке, top джоб по диску и времени, топ payload по размеру. Их задача, быстро показать 20 процентов причин, которые дают 80 процентов нагрузки. Это превращает оптимизацию в прицельную работу, а не в общий рефакторинг.

Пятый тип, "Forecast и сценарии". Это прогноз нагрузки и прогноз headroom на горизонте 4-12 недель с доверительными интервалами. Плюс сценарии, что будет при росте на 20 процентов, что будет при включении новой фичи, что будет при изменении retention или объема данных. Для решения, когда нужно закупать ресурсы, когда запускать Capacity инициативы и где risk window. В идеале в отчете есть даты, когда headroom уходит ниже порога, и какие инициативы нужны, чтобы отодвинуть эту дату.

Шестой тип, "Capacity incidents и near misses". Это аналитика по инцидентам и почти-инцидентам. Сколько раз за период мы приближались к насыщению, сколько было emergency скейлов, rate limit, деградаций. Доля времени в красной зоне. Это оценка зрелости процессов, если таких событий меньше и они предсказуемее, значит мониторинг и планирование работают.

Седьмой тип, "Efficiency of portfolio", отчет по эффективности Capacity инициатив. Для каждой инициативы есть baseline и подтвержденный эффект, например изменение capacity на целевом SLO, изменение unit cost, изменение headroom. Это отчет для управления портфелем, он помогает отсеивать инициативы, которые не дают эффекта, и усиливать те, которые дают.

По инструментам я держу минимальный стек. Дашборды в Grafana, метрики в Prometheus, трассы в OpenTelemetry с Jaeger или Tempo, логи в ELK или аналогах, метрики БД из pg_stat_statements, wait events, slow query log, для ClickHouse system.query_log. Для отчетов, которые нужны руководству, делаю короткие регулярные срезы, но источником правды остается дашборд с базовыми метриками и одинаковыми определениями.

## Как вы будете обеспечивать достоверность данных, используемых для принятия решений в области Capacity Management?
<a id="Ensuring-Data-Integrity-for-Decision-Making-in-Capacity-Management"></a>
([наверх](#sections))

Достоверность данных в Capacity Management я обеспечиваю так, чтобы любая цифра в дашборде отвечала на три вопроса. Что именно измеряем. Насколько это измерение полно и стабильно. Можно ли это воспроизвести и проверить альтернативным источником.

**1) Единые определения и владельцы метрик**

Первое, фиксирую словарь метрик. Для каждой метрики есть определение, единицы измерения, окно агрегации, правила фильтрации, обязательные разрезы, например endpoint, tenant, статус ответа. У метрики есть владелец, который отвечает за семантику и изменения. Без этого одна команда считает RPS по входящим запросам, другая по успешным, третья по upstream, и портфель решений начинает строиться на несопоставимых числах.

**2) Контракты на телеметрию и контроль изменений**

Инструментация меняется так же управляемо, как код. Я ввожу telemetry contracts. Какие события обязаны быть, какие поля обязаны быть, какие значения допустимы, какие лейблы запрещены из-за кардинальности. Любое изменение схемы метрик и логов проходит через review, иначе внезапный релиз может обнулить ряд, поменять единицы или разрезы, и прогнозы станут мусором.

**3) Надежность пайплайна данных и SLO на наблюдаемость**

Для метрик и логов я считаю критичным иметь собственные SLO. Задержка доставки, процент потерь, доля пропусков, объем дропа из-за лимитов, стабильность кардинальности. Если система наблюдаемости деградирует, любые решения по капасити становятся рискованными. Поэтому я делаю мониторинг самого мониторинга, алерты на пропадание серий, на резкие скачки кардинальности, на рост lag в очередях.

**4) Кросс-проверки между независимыми источниками**

Ключевой прием, сверять одно и то же явление по разным каналам. Примеры, которые реально ловят ошибки

* RPS по счетчику в приложении сравниваю с RPS по ingress или балансировщику, расхождение выше порога означает проблему в инструментировании или в фильтрах
* latency из приложения сверяю с latency на уровне прокси, сильные расхождения часто означают измерение только части пути
* CPU и память по cgroup приложения сверяю с node exporter на хосте, это быстро выявляет noisy neighbor или неверные лимиты
* запросы к базе по метрикам приложения сверяю с pg_stat_statements или query log, это ловит недосбор, sampling и ошибки тегирования

Важно, чтобы эти сверки были регулярными и автоматическими, не раз в квартал руками.

**5) Временные метки, окна агрегации, late events**

Для Capacity очень важна корректная ось времени. Я привожу все к единой таймзоне, синхронизирую часы на хостах, явно выбираю, где используется event time, где ingestion time. Для логов и событий обрабатываю late arrivals и дубликаты. Иначе пики расползаются, p99 начинает выглядеть лучше или хуже, чем есть, а headroom считается неправильно.

**6) Репрезентативность, sampling и смещения**

Если есть sampling, я фиксирую коэффициенты и метод. Что сэмплируется, по каким правилам, можно ли восстановить оценку без смещения. Частая проблема, семпл по ошибкам или по медленным запросам, после чего средние становятся нерелевантными. Для капасити я обычно оставляю критичные метрики без sampling, а тяжелые детали, например тела запросов, сэмплирую отдельно.

**7) Воспроизводимость расчетов и версионирование дашбордов**

Любой расчет, который используется для решений, должен быть воспроизводим. Я храню запросы, преобразования и версии дашбордов. Если поменялась формула, остается история и дата изменения. Это убирает ситуацию, когда на прошлой неделе headroom был 30%, а сегодня стал 10% из-за правки агрегации.

**8) Проверка на уровне нагрузочных тестов и продакшена**

Результаты синтетических тестов я считаю достоверными, когда они калиброваны на проде. То есть профиль запросов совпадает, ключевые метрики стоимости запроса сопоставимы, поведение хвостов похоже. Если тестовый стенд показывает одно, а прод другое, значит данные теста непригодны для принятия решений о закупке или о переносе лимитов, пока не найдена причина расхождения.

**9) Доступы и аудит**

Данные для Capacity влияют на бюджеты и риск инцидентов, поэтому я ограничиваю право менять источники, фильтры и базовые запросы. Нужен аудит изменений, кто поменял, когда, зачем, какая ссылка на тикет. Это дисциплинирует и защищает от случайных правок.

В итоге доверие к данным держится на двух опорах. Четкие определения и управляемые изменения. Автоматические проверки, которые ловят расхождения раньше, чем по этим данным примут неправильное решение.


# Технические знания
<a id="Technical-Knowledge"></a>
([наверх](#sections))

## Какие основные компоненты составляют серверную инфраструктуру в дата-центрах?
<a id="Key-Components-of-Server-Infrastructure-in-Data-Centers"></a>
([наверх](#sections))

Серверная инфраструктура в ДЦ это несколько слоев, которые вместе дают вычисления, хранение, сеть, управление и эксплуатационную надежность. Удобно мыслить не отдельными железками, а цепочками, где у каждого слоя есть свои точки отказа, лимиты и инструменты контроля.

**1) Физический слой размещения**

Основа, где все стоит и как соединено.

[Рэки](/DictionaryOfTerms.md#рэки-в-дата-центрах), направляющие, кабельная инфраструктура, [кроссы](/DictionaryOfTerms.md#кроссы-в-дата-центрах), патч-панели, маркировка. От этого зависит скорость работ, вероятность ошибок при перекоммутации и способность выдерживать плотность по питанию и охлаждению.

**2) Вычислительный слой**

То, что обычно называют compute.

Серверы разных ролей, чаще всего это узлы под приложения, базы данных, аналитические движки, инфраструктурные сервисы. Внутри важны [CPU](/DictionaryOfTerms.md#cpu), [RAM](/DictionaryOfTerms.md#ram), локальные диски, [PCIe устройства](/DictionaryOfTerms.md#pcie-устройства), а также [out-of-band управление через BMC](/DictionaryOfTerms.md#out-of-band-управление-через-bmc). Типичный контур управления, [IPMI](/DictionaryOfTerms.md#ipmi) или [iDRAC/iLO](/DictionaryOfTerms.md#idracilo) плюс [KVM over IP](/DictionaryOfTerms.md#kvm-over-ip), чтобы можно было восстановить узел даже при проблемах с ОС и сетью.

**3) Сетевой слой**

Связность внутри ДЦ и наружу.

На практике это сеть уровня стойки и сеть агрегации. Снизу [ToR коммутаторы](/DictionaryOfTerms.md#tor-коммутаторы) в стойках, выше [spine-leaf фабрика](/DictionaryOfTerms.md#spine-leaf-фабрика), далее [border](/DictionaryOfTerms.md#border-в-сети-дата-центра), [NAT](/DictionaryOfTerms.md#nat), [DDoS](/DictionaryOfTerms.md#ddos), [внешние пиринги](/DictionaryOfTerms.md#внешние-пиринги) и каналы. Внутри этого слоя обычно живут L4 балансировщики, иногда L7, а также сервисы маршрутизации, [VRF/VLAN сегментация](/DictionaryOfTerms.md#vrf-и-vlan-сегментация), [ACL](/DictionaryOfTerms.md#acl).

Если говорить про капасити, сеть в ДЦ это не только гигабиты, но и [устойчивость по pps](/DictionaryOfTerms.md#устойчивость-по-pps), буферы, [oversubscription](/DictionaryOfTerms.md#oversubscription), [ECMP распределение](/DictionaryOfTerms.md#ecmp-распределение), потери и [ретрансмиты](/DictionaryOfTerms.md#ретрансмиты).

**4) Хранилище**

Где лежат данные и как они доступны.

Есть локальные диски в серверах и есть сетевое хранилище. Сетевое может быть [SAN](/DictionaryOfTerms.md#san), [NAS](/DictionaryOfTerms.md#nas) или распределенные системы, которые собирают данные из дисков множества узлов. Отдельно стоит объектное хранилище как слой для бэкапов, архивов, data lake. Для хранилища важны контроллеры, [RAID](/DictionaryOfTerms.md#raid) или [erasure coding](/DictionaryOfTerms.md#erasure-coding), политика репликации, а еще отдельная сеть под storage traffic, если она выделяется.

**5) Питание и электрическая цепочка**

Слой, который часто является настоящим источником ограничений по плотности.

В цепочке обычно есть вводы, [ИБП](/DictionaryOfTerms.md#ибп), генераторы, распределение по линиям, [PDU](/DictionaryOfTerms.md#pdu) в стойках, мониторинг потребления. Критичная часть, резервирование, схема N+1 или 2N, и реальные лимиты на стойку, например 5-10 кВт, которые определяют, сколько серверов можно поставить и какой класс CPU допустим без [троттлинга](/DictionaryOfTerms.md#троттлинг).

**6) Охлаждение и климат**

То, что держит железо в расчетных режимах.

[CRAC/CRAH](/DictionaryOfTerms.md#crac-и-crah), горячие и холодные коридоры, [containment](/DictionaryOfTerms.md#containment), датчики температуры и влажности, иногда жидкостное охлаждение для высокой плотности. Этот слой напрямую влияет на стабильность частот CPU и на ресурс дисков. Для эксплуатации важно, что тепловые проблемы часто выглядят как деградация производительности без явных ошибок.

**7) Безопасность**

Физическая и логическая.

Физическая это доступ в залы, охрана, видеонаблюдение, контроль доступа по зонам, журналирование посещений. Логическая это сегментация сети, межсетевые экраны, bastion, политики доступа к управлению, управление секретами и ключами.

**8) Управляющая и платформенная инфраструктура**

То, что делает ДЦ пригодным для регулярной эксплуатации.

Сюда входят системы виртуализации, если она используется, гипервизоры и управляющие кластеры. Плюс контейнерная оркестрация, если основная платформа на Kubernetes, а также registry, CI/CD интеграции, service discovery, DNS, [PKI](/DictionaryOfTerms.md#pki), [NTP](/DictionaryOfTerms.md#ntp). В этот же слой обычно входят балансировка, [ingress](/DictionaryOfTerms.md#ingress), прокси, политики безопасности, control plane наблюдаемости.

**9) Наблюдаемость и эксплуатационные процессы**

Без этого железо есть, управляемого капасити нет.

Мониторинг метрик, логирование, трассировка, алертинг, инвентаризация, [CMDB](/DictionaryOfTerms.md#cmdb), управление конфигурациями, автоматизация выката, процедуры аварийного восстановления. Здесь же плановое обслуживание, замены дисков, обновления прошивок, управление спэрами, контроль SLA поставщиков.

## Какие основные технологии используются для масштабирования кластеров и расширения капасити?
<a id="Core-Technologies-for-Scaling-Clusters-and-Expanding-Capacity"></a>
([наверх](#sections))

Если упростить, расширение капасити делается двумя путями. Добавляем ресурсы, то есть узлы, диски, каналы. Меняем способ распределения работы, чтобы тот же ресурс давал больше полезного throughput. Технологии ниже как раз про это, как быстро добавлять, как правильно распределять, как не сломать состояние и как удержать SLO.

**1) Оркестрация и управление вычислениями**

Главная технология горизонтального масштабирования сегодня это оркестратор, который умеет размещать нагрузки по узлам и переживать отказы.

Kubernetes, реже Nomad, Mesos в старых установках. Что именно дает для капасити

* быстрый scale out по репликам сервиса и rolling update
* bin packing, то есть более плотное и осмысленное размещение по CPU и RAM
* автоматическое восстановление и перераскладывание после падений узлов
* контроль ресурсов через requests and limits, чтобы избежать noisy neighbor

Автомасштабирование

* HPA, масштабирование по метрикам, например RPS, latency, custom metrics
* VPA, подстройка запросов ресурсов
* Cluster Autoscaler, добавление или удаление нод в кластере

Практичный смысл такой, при росте нагрузки cap кластера растет почти линейно с числом узлов, пока не упрется в состояние или сеть

$$Capacity_{cluster} \approx k_{eff}\sum_i Capacity_i$$

Где $k_{eff}$ это потери на координацию, балансировку, фоновые задачи и неидеальную упаковку.

**2) Распределение трафика и балансировка**

Чтобы новые узлы реально давали прирост, трафик должен распределяться правильно.

Технологии уровня L4 и L7

* L4 балансировщики, IPVS, LVS, cloud LB, Envoy в режиме L4
* L7 ingress, Nginx, Envoy, HAProxy, cloud ALB, API Gateway

Технологии маршрутизации между сервисами

* service discovery, DNS, Consul, Kubernetes Service
* service mesh, Istio, Linkerd, когда нужны политики трафика и наблюдаемость

Полезные механики для капасити

* weighted routing и canary, чтобы добавлять капасити постепенно
* rate limiting и backpressure на входе, чтобы не убивать кластер пиками
* circuit breaker, чтобы не получить шторм ретраев

**3) Масштабирование stateful систем, репликация, шардинг, партиционирование**

Вычисления масштабировать проще, состояние сложнее. Поэтому ключевые технологии капасити обычно живут вокруг данных.

Репликация

* read replicas, реплики под чтение в Postgres, MySQL, MongoDB
* multi-replica в распределенных БД, Cassandra, ClickHouse ReplicatedMergeTree

Шардинг

* диапазонный, по ключу, по tenant
* consistent hashing, чтобы при добавлении узлов перераспределять минимум ключей
* роутинг запросов на шарды через прокси, например PgBouncer плюс логика в приложении, или специализированные прокси

Партиционирование

* time-based партиции для больших таблиц
* hash-partitioning, чтобы равномерно разложить нагрузку
* TTL и tiering, чтобы ограничивать рабочий набор и снижать давление на диск

Материализация и предагрегаты

* материализованные представления, precomputed витрины, агрегаты
* это технология расширения капасити через снижение стоимости запроса, а не через добавление железа

**4) Распределенное хранилище и расширение дисковой емкости**

Для хранения важны технологии, которые позволяют добавлять диски и узлы без остановки сервиса и с автоматическим ребалансом.

Типовые варианты

* distributed storage, Ceph, MinIO, HDFS, GlusterFS
* object storage, S3 API как слой для lake и бэкапов
* erasure coding и репликация на уровне хранилища

Ключевая технология здесь это online rebalancing. Добавили новые узлы, данные постепенно перетекают, сервис продолжает работать, пользователь видит только временное изменение latency, если ребаланс не ограничен по bandwidth.

**5) Масштабирование очередей и стриминга**

В системах с событиями капасити часто ограничивается брокером и потребителями.

Kafka и аналоги

* масштабирование через partitions, увеличение параллелизма консюмеров
* балансировка consumer group, ребаланс, sticky assignment
* отдельный throughput на topic и partition, поэтому проектирование количества партиций это заранее закладываемый капасити

RabbitMQ, NATS JetStream и похожие

* кластеризация и sharding по очередям
* подтверждения, дедупликация, ретраи, чтобы нагрузка не взрывалась при сбоях

Для вычислительных фреймворков, Spark, Flink, Ray, ключевые технологии это autoscaling воркеров, state backend, checkpointing. Они позволяют расширять капасити без потери состояния и без полного рестарта вычисления.

**6) Технологии расширения капасити без остановки**

Это не отдельные продукты, это классы техник, которые почти всегда нужны.

* rolling deployments, чтобы менять версии без даунтайма
* backward compatible миграции схем, expand then migrate then contract
* dual write и shadow read, когда нужно безопасно перейти на новую схему хранения или новый кластер
* throttling миграций и ребаланса, чтобы расширение капасити не съело весь диск или сеть

**7) Наблюдаемость и контроль как часть технологии масштабирования**

Без измерений масштабирование превращается в гадание. Поэтому в реальном стеке рядом всегда

* метрики, Prometheus, Grafana
* трассировка, OpenTelemetry
* логи и slow query log, профили, eBPF в проде по необходимости

Практическая цель, видеть, что после добавления узлов $Capacity$ реально вырос, а $k_{eff}$ не ухудшился из-за новых узких мест, например ребаланс данных начал грузить диск, или балансировщик стал ограничением.

## Как вы оцените пропускную способность сети и примените эту информацию к управлению капасити?
<a id="Evaluating-Network-Capacity-and-Applying-This-Information-to-Capacity-Management"></a>
([наверх](#sections))

# Оценка пропускной способности сети для управления капасити

Я делаю это в два прохода. Сначала считаю теоретический потолок и проверяю, реалистичен ли он с учетом накладных расходов. Потом меряю фактический потолок в проде и нахожу, где именно он теряется, линк, pps, очереди, ретрансмиты, балансировка, oversubscription.

**1) Что именно считаю "пропускной способностью"**

Для capacity важны три разные величины, и они часто ограничивают по очереди.

1. Throughput по байтам, сколько полезных байт в секунду можно передать
2. PPS, сколько пакетов в секунду выдерживают NIC, хост и сеть
3. Латентность и хвосты, потому что сеть может "держать гигабиты", но ломать p99 из-за очередей и микробёрстов

Если смотреть только на гигабиты, можно ошибиться, когда трафик мелкими пакетами и упор в pps, или когда ретрансмиты и очереди съедают полезную полосу.

**2) Теоретический потолок, быстрый расчет**

Беру скорость линка $R_{link}$ в бит/с и перевожу в байты в секунду. Для 10GbE это $10 \cdot 10^9$ бит/с, делю на 8, получаю 1.25 GB/s на "проводе". Дальше учитываю эффективность $\eta$, она зависит от MTU, протокола, шифрования, количества hop и качества линии.

Формула для полезной полосы такая

$$BW_{eff} \approx \frac{R_{link}}{8} \cdot \eta$$

На практике для TCP с MTU 1500, без сильных потерь, $\eta$ часто оказывается порядка 0.85-0.95. С jumbo frames $\eta$ растет, при маленьких пакетах падает.

Отдельно считаю pps потолок. Для MTU 1500 можно прикинуть, сколько пакетов нужно для заданного throughput. Приближенно

$$PPS \approx \frac{BW_{eff}}{Payload_{avg}}$$

Где $Payload_{avg}$ это средний полезный размер пакета в байтах. Если payload маленький, pps резко растет, и ограничение может стать именно pps, даже когда байтовая полоса далека от насыщения.

Пример, линк 10GbE, пусть $BW_{eff}$ около 1.1 GB/s. Если средний payload 300 байт, то pps порядка 3.7 млн пакетов/с. Дальше уже важны возможности NIC, драйвера, RSS, IRQ affinity, настройки coalescing, и способность сетевой фабрики держать такую частоту без потерь.

**3) Фактический потолок, как меряю в реальности**

Я не полагаюсь на один источник, сверяю минимум два уровня.

На хосте
интерфейсные счетчики rx/tx bytes, rx/tx packets, drops, errors, tcp retransmits, softnet drops, iowait на сетевые прерывания. Это показывает, упираемся ли мы в NIC, в CPU на сетевом стеке, или в очередь.

На сетевом оборудовании
интерфейсы ToR и uplink, utilization, queue depth, ECN, drops на портах, oversubscription по стойке, распределение по ECMP. Это показывает, не обманывает ли нас "красивый" график на хосте, когда узкое место на аплинке или в агрегации.

Плюс активная проверка, когда нужно подтвердить цифру
iperf3 или аналог для point-to-point, и обязательно в обе стороны. Для реальной картины полезно проверять несколько параллельных потоков, потому что один TCP поток часто не добирает полосу из-за окна и латентности.

Очень важная часть, смотреть на потери и ретрансмиты. Даже доли процента потерь на высоких скоростях могут убить полезный throughput и хвосты latency.

**4) Как перевожу сетевой потолок в капасити сервиса**

Дальше делаю то, что удобно для управления, считаю "сетевую цену" запроса. Пусть один входной запрос вызывает в среднем $k$ downstream вызовов, а суммарный трафик на один входной запрос $B_{net_{req}}$ байт, туда входят request/response и межсервисный трафик.

Тогда оценка потолка по сети такая

$$RPS_{net} \approx \frac{BW_{eff}}{B_{net_{req}}}$$

Если еще и упор в pps, то считаю среднее число пакетов на запрос $p_{req}$ и потолок по pps

$$RPS_{pps} \approx \frac{PPS_{max}}{p_{req}}$$

И реальный сетевой потолок для сервиса это минимум из этих двух, плюс запас на пики и микробёрсты.

Здесь важно разделять north-south и east-west. В микросервисной архитектуре east-west часто доминирует, поэтому $B_{net_{req}}$ должен учитывать межсервисные hops, иначе оценка будет слишком оптимистичной.

**5) Как применяю это в Capacity Management**

Дальше сеть становится нормальным ресурсом в модели, как CPU или диск.

1. Headroom по сети
   беру текущий $Load_{net}$ и оцениваю запас $1 - \frac{Load_{net}}{BW_{eff}}$. Если запас падает ниже порога в пиковые окна, это триггер для инициатив.

2. Прогноз
   если прогнозируется рост RPS или рост $B_{net_{req}}$ из-за новой фичи, я заранее пересчитываю $RPS_{net}$ и вижу дату, когда потолок будет достигнут.

3. Выбор стратегии расширения
   если упор в байты, помогают сжатие, уменьшение payload, более компактные форматы, кеширование ближе к потребителю, сокращение hops, bulk API.
   если упор в pps, помогают укрупнение сообщений, batching, уменьшение чата между сервисами, правильная настройка RSS и распределение прерываний, иногда повышение MTU.
   если упор в хвосты из-за очередей, помогают shaping, приоритизация трафика, защита от ретраев, снижение микробёрстов, проверка oversubscription и ECMP.

4. Планирование работ, которые сами съедают сеть
   ребаланс шардов, репликация, бэкапы, миграции. Их я считаю отдельной нагрузкой на $BW_{eff}$. Если пустить их в пик, можно "съесть" headroom и получить деградацию пользовательского трафика.

**6) Типовые ловушки, которые я проверяю всегда**

Первая, линк 10G есть, а фактический $BW_{eff}$ вдвое ниже из-за CPU на сетевом стеке, шифрования, неправильного RSS или малого числа очередей NIC.

Вторая, внутри стойки все хорошо, а аплинк ToR в spine перегружен из-за oversubscription, и проблема проявляется только на межстоечных запросах.

Третья, throughput нормальный, p99 плохой. Обычно причина в очередях, микробёрстах или ретрансмитах, и это не лечится добавлением "еще 10 процентов полосы", пока не исправишь источник потерь и поведение трафика.

# Балансировщик нагрузки
<a id="Load-Balancer"></a>

## Зачем между приложением и вебсерверами нужен балансировщик нагрузки в контексте систем дизайна?
<a id="Why-is-a-Load-Balancer-Needed-Between-Application-and-Web-Servers-in-System-Design-Context"></a>
([наверх](#sections))

Балансировщик нагрузки между клиентским трафиком и пулом веб серверов нужен, чтобы превратить набор отдельных машин в один управляемый, масштабируемый и отказоустойчивый слой. В терминах систем дизайна он решает сразу несколько задач, которые иначе пришлось бы размазывать по клиентам, DNS, приложению и ручным операациям.

**Единая точка входа и масштабирование без изменений у клиентов**

Клиенту нужен один стабильный адрес. Серверов при этом может стать 3, потом 30, часть уйдет на обновление, часть умрет, часть добавится под пик. Балансировщик скрывает динамику пула, клиент не должен знать, сколько сейчас инстансов и где они.

Если упрощать модель капасити веб слоя, то без перекосов она близка к сумме

$$Capacity_{web} \approx k_{eff}\sum_i Capacity_i$$

Где $k_{eff}$ отражает потери на неидеальное распределение, очереди, warmup, разный перформанс нод. Балансировщик как раз уменьшает эти потери, он снижает перекосы и быстрее утилизирует добавленные узлы.

**Отказоустойчивость и быстрый фейловер**

В реальности сервера ломаются постоянно, процесс упал, JVM в стопе из-за GC, диск в iowait, сеть деградировала, нода перегрелась и троттлит. Балансировщик делает health checks и исключает плохие инстансы из раздачи трафика. Это резко уменьшает шанс, что пользователи попадут на мертвую ноду и будут ждать таймаут.

Важно, что health check должен быть близок к пользовательской реальности. Если проверять только “порт открыт”, можно продолжать слать трафик на сервер, который уже не отвечает по бизнес логике.

**Управление латентностью и очередями**

Даже когда все “живы”, сервера бывают в разных состояниях. Один прогрет, другой только поднялся, третий под нагрузкой держит меньшее число соединений. Балансировщик может выбирать стратегию распределения, round robin, least connections, least response time, consistent hashing. Это позволяет держать хвосты latency в узде.

Плюс балансировщик это место, где удобно включать ограничения и backpressure. Когда входной поток $\lambda$ превышает устойчивую производительность, очередь растет по закону Литтла

$$L = \lambda \cdot W$$

Если продолжать принимать все подряд, $W$ начинает раздуваться, таймауты и ретраи создают шторм. Балансировщик позволяет раньше отрезать лишнее, например 429, и сохранить SLO для тех, кого обслужить реально.

**Безопасные релизы и изменение топологии**

В систем дизайне почти всегда нужен контролируемый выкат. Canary, blue green, gradual rollout, weighted routing, быстрый rollback. Без балансировщика это превращается в ручную магию с DNS и клиентами.

Балансировщик также помогает при миграциях. Можно постепенно уводить трафик на новый кластер, сравнивать ответы, держать часть запросов в shadow режиме. Это прямо про управление рисками при расширении капасити.

**TLS, безопасность и политики трафика**

На практике балансировщик часто берет на себя termination TLS. Это упрощает управление сертификатами и снижает CPU нагрузку на веб серверах, особенно при большом числе коротких соединений. Там же удобно централизованно применять WAF, rate limiting, ограничения по IP, защита от простых DDoS сценариев, нормализация заголовков, ограничения по размеру запроса.

Это важно, потому что “чисто сеть” и “чисто безопасность” напрямую влияют на капасити. Один неудачный ретрай или шторм соединений может съесть больше ресурсов, чем реальная бизнес нагрузка.

**Наблюдаемость и контроль поведения клиентов**

Балансировщик дает единый слой, где видно поток запросов, ошибки, латентность, размеры ответов, повторные попытки, перекосы по upstream. Это быстрый способ понять, проблема в одном инстансе, в конкретном endpoint, в клиенте, в сети, или в downstream.

Без этого многие проблемы выглядят как “все медленно”, пока ты не увидишь, что 5 процентов запросов дают 80 процентов таймаутов и летят в один сегмент пула.

**Почему не хватает DNS round robin**

DNS распределение плохо подходит для систем дизайна, где важны быстрый фейловер и управляемые релизы. Есть кеширование на клиентах и резолверах, TTL работает не так предсказуемо, часть клиентов продолжит ходить на плохой IP. Плюс DNS не знает, какой инстанс перегружен, какой деградирует, и не умеет тонко управлять весами на уровне процентов трафика.

**Важные нюансы, чтобы балансировщик не стал проблемой**

Балансировщик сам может стать узким местом или SPOF, поэтому его делают отказоустойчивым, несколько экземпляров, anycast или active active, разнесение по зонам. Еще заранее решают вопрос с сессиями. Если есть sticky sessions, это упрощает stateful логику, но ухудшает равномерность распределения и усложняет масштабирование. Поэтому часто стремятся к stateless на веб слое, а состояние выносить в отдельные хранилища.

Итог такой. Балансировщик в систем дизайне это не “еще одна коробка”, это механизм, который превращает пул веб серверов в управляемый ресурс. Он повышает доступность, увеличивает полезную утилизацию капасити, упрощает релизы и делает поведение системы предсказуемее под ростом нагрузки.
